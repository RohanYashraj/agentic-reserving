# Story 5.3: Method Recommendations Through the Gate

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an Analyst,
I want a per-Origin-Period Method recommendation with reasons citing Diagnostic IDs,
so that the layer's judgment is pinned to evidence I can inspect. (FR-10)

## Acceptance Criteria

1. **Given** a complete Run with Diagnostics, **When** the agent generates recommendations, **Then** every Origin Period in the Run receives exactly one recommended Method with ≥1 reason, each reason citing ≥1 Diagnostic ID resolvable against this Run, validated programmatically before output is accepted (FR-10).
2. **And** an output missing an Origin Period, doubling one, or citing an unresolvable ID is rejected and retried within the bounded loop; persistent failure surfaces as a failed Interpretation, never partial output.
3. **Given** the accepted output, **When** it is returned to Convex, **Then** it persists as a typed recommendations document linked to the Run, with the full LLM transcript appended to the Audit Log (FR-9, FR-15).
4. **And** the recommendation draft is run through the Provenance Gate (Story 5.2) — every figure a `{{rs:...}}` placeholder, every quantitative claim carrying a resolvable `{{dx:...}}` citation — before it is ever persisted or returned as accepted (AD-5, AD-1).

- [x] **Task 4: Bounded generate-gate-validate loop — new `engine/engine_service/recommendations_flow.py` (AC: 1, 2, 4)**
  - [x] 4.1 This is the imperative composition the deferred-work from 5.1/5.2 pointed here: it wires `copilot_agent` (agent + tools + transcript) + `provenance_gate` (5.2) + `validate_recommendations` (Task 1) into one bounded loop.   - [x] 4.2 **The bounded redraft loop lives HERE, in engine_service — NOT in Convex** (this reverses the 5.2 deferred note's tentative "the caller drives the loop"; see Dev Notes §The redraft loop belongs in engine_service for the rationale). Signature roughly:
    ```python
    def generate_recommendations(
        model: Model,
        result_set: ResultSet,
        diagnostics_bundle: DiagnosticsBundle,
        *,
        max_attempts: int = 3,
    ) -> RecommendationsOutcome
    ```
    where `model` is INJECTED (the 5.1 test seam — tests pass a scripted `Model`, prod passes `build_gemini_model(settings.gemini_api_key, settings.gemini_model_id)`). `RecommendationsOutcome` is a typed discriminated result carrying, in BOTH arms, the per-attempt transcripts (for audit — FR-15): `RecommendationsAccepted(recommendations: Recommendations, attempts: tuple[AttemptRecord, ...])` | `RecommendationsFailed(reason_summary: ..., attempts: tuple[AttemptRecord, ...])`. `AttemptRecord(transcript: Transcript, rejections: tuple[...])` records what the model produced and why it was rejected (empty rejections on the accepted attempt).
  - [x] 4.3 Each loop iteration:
    1. Build a fresh agent per attempt: `build_interpretation_agent(model, result_set, diagnostics_bundle, instructions=build_recommendation_prompt(...))`; on a redraft, append the prior attempt's rejection reasons to the prompt (the model is told exactly what failed — missing origin X, unresolvable citation Y, unsourced number Z — so it can fix it). No durable Agno state (AD-3) — the redraft context is re-supplied via the prompt, not a persisted session.
    2. `result = run_interpretation(agent, prompt)`; capture `result.transcript` for this attempt regardless of outcome.
    3. `parse_recommendation_draft(result.output_text)` → on `DraftParseError`, record the attempt with a `draft_unparseable` rejection and continue the loop.
    4. **Gate each reason** through `run_provenance_gate(reason_text, result_set, diagnostics_bundle)` (5.2). A `GateRejected` on any reason → collect its `GateRejection`s into this attempt's rejections. A `GateAccepted` gives the rendered `reason.text` + `reason.citations` (the resolved dx ids) — assemble the candidate `RecommendationReason`. (This is AC-4: the draft never becomes accepted content without passing the gate; a figure not written as `{{rs:...}}` fails `unsourced_number`, an uncited quantitative reason fails `uncited_claim`.)
    5. Assemble a candidate `Recommendations` from the gated reasons + `run_id = diagnostics_bundle.run_id`, then `validate_recommendations(candidate, result_set, diagnostics_bundle)` (Task 1.3) for the STRUCTURAL guarantees the per-reason gate can't see (coverage: every origin exactly once; method actually run). Combine gate rejections + structural rejections into the attempt's rejection set.
    6. If the attempt has **zero** rejections → `RecommendationsAccepted(candidate, attempts=...)`, return immediately. Else record the `AttemptRecord` and loop.
  - [x] 4.4 **Exhaustion (AC-2): after `max_attempts` with no clean attempt → `RecommendationsFailed(...)`** carrying every attempt's transcript + rejections. NEVER return partial output — a `RecommendationsFailed` has no `recommendations` attribute (structural never-persist guarantee, mirroring `GateRejected`). This is the "persistent failure surfaces as a failed Interpretation, never partial output" clause.
  - [x] 4.5 **Model-not-configured is a distinct, non-retryable outcome (Engine-Only Mode seam, AD-9).** `build_gemini_model` raises `ModelNotConfiguredError` when the key/id are absent — do NOT catch it inside the loop as a redraftable rejection; let it propagate to the route (Task 5), which maps it to a typed `model_unavailable` failure. Story 5.6 derives Engine-Only Mode from this; 5.3 only surfaces it cleanly (do not swallow, do not retry a missing key). The per-Run token/cost **ceiling** (the other AD-9 trigger + the real value of `max_attempts`/an interpretation timeout) stays deferred to 5.6 — here `max_attempts` is a plain module constant with a comment pointing at 5.6.
  - [x] 4.6 Keep `generate_recommendations` deterministic given a deterministic (scripted) model — no clock, no randomness, no logging — so it is golden-testable with the 5.1 `_ScriptedModel` idiom. The only nondeterminism in prod is the live model, isolated behind the injected `model` param.

## Dev Notes

### Architecture compliance (non-negotiable)

- **AD-1 / The Constitution — the LLM never calculates.** Recommendations carry method choices + textual reasons + Diagnostic-ID citations; any figure a reason states is written as a `{{rs:...}}` placeholder the gate renders from the ResultSet, never a literal the model computed. `validate_recommendations` and the loop compute NO reserve figures (no sums/deltas/ratios). The recommendations document is machine-drafted content governed by the gate (AD-5) — persist only what passed.
- **AD-5 — Provenance Gate governs machine-drafted content.** Every reason runs through `run_provenance_gate` (5.2) before it can become an accepted `RecommendationReason`: unsourced numbers → `unsourced_number`, uncited quantitative claims → `uncited_claim`, unresolvable citations → `unresolvable_dx_citation`. A draft that fails is never returned/persisted as reviewable; the rejection (with the transcript) is audit-logged and drives the bounded redraft. Prompt instructions (Task 3) are NOT a substitute for the programmatic gate — they only make the model more likely to pass it.
- **AD-8 — read-only agent over Agno/google-genai.** The agent reaches evidence ONLY through the four read-only tools (5.1); the recommendation prompt is provider-neutral; the single provider seam is `build_gemini_model`. Model id from `engine_service` config. No new tool, no write, no network beyond the model plane.
- **AD-3 — Convex is the sole system of record; engine_service is stateless & never calls Convex.** The Run's ResultSet + DiagnosticsBundle are passed DOWN into `/recommendations` (not fetched by the engine); the accepted Recommendations + the transcript are returned UP to the Convex action, which persists + audits. No durable Agno session state — redraft context is re-supplied via the prompt, not a stored session.
- **AD-6 / FR-15 — every LLM interaction is audit-logged.** Each attempt's full transcript (prompt, tool calls/results, response) rides in the audit payload — on success (`run.recommended`) AND on each rejection (`run.interpretationRejected`). `appendAuditEntry` is the sole audit writer; the chain stays append-only + hash-chained.
- **AD-4 — no anonymous Convex access.** `generateRecommendations` / `getRecommendations` call `requireMember` first. Override (senior_actuary) is Epic 6, not here.
- **AD-9 — fails closed into Engine-Only Mode.** `ModelNotConfiguredError` → typed `model_unavailable` (not retried, not swallowed); Story 5.6 derives Engine-Only Mode from it. The per-Run cost ceiling + interpretation timeout are 5.6's; here `max_attempts` is a plain constant.
- **AD-10 — camelCase cross-runtime contract, drift-checked.** The `Recommendations` model is schema-exported and its Convex validator is drift-checked in CI exactly like ResultSet/DiagnosticsBundle. Diagnostic IDs are resolved via `resolve_diagnostic`, never string-split.

### The redraft loop belongs in engine_service (reversing the 5.2 deferred note)

The 5.2 deferred note tentatively said "the caller (a Convex action) drives the bounded redraft loop." **Put the loop in engine_service instead**, and here is why: a redraft is an *agent turn* that must see what failed and try again. Keeping the loop in-process (Task 4) lets each attempt re-prompt the same model within one request, preserves the transient Agno session semantics (AD-3), and keeps the gate + agent co-located in the shell that hosts them (AD-2/AD-8). If Convex drove the loop, every attempt would be a fresh HTTP round-trip and a fresh agent session, and Convex would have to thread the rejection reasons back into the next engine call — more moving parts, more audit plumbing, and Convex reaching into model-plane concerns. With the loop in engine_service, the Convex action stays the thin persist+audit tail it is for `/runs` and `/rederive` (`rederiveRun` is the exact template). Both designs audit the same data (every attempt's transcript + rejections are returned to Convex either way); this one is simpler and better-layered. Documented as a deliberate reversal so a reviewer isn't surprised.

### Persistence shape — inline on `runs`, not a child table

Every engine artifact in this codebase is persisted as an optional field on the `runs` row (`resultSet`, `diagnosticsBundle` — `schema.ts:130-131`), read back verbatim by a `getX` query twin (`getResultSet`/`getDiagnosticsBundle`), written by a status-guarded `storeX` internalMutation (`storeResultSet`). Follow that exact pattern for `recommendations`: one optional field, one `storeRecommendations`, one `getRecommendations`. A separate `recommendations` table keyed by `runId` would introduce the repo's first child-of-runs table with no precedent — avoid the new pattern. The "typed recommendations document linked to the Run" is satisfied by the row: the recommendations ARE on the Run. (If Epic 6 later needs recommendations versioned independently of the Run — e.g. draft vs overridden — revisit then; flagged in deferred-work.)

### Direct action vs durable workflow

`createRun` uses `@convex-dev/workflow` because the engine `/runs` call is part of a durable, retry-on-transient-failure orchestration with `queued→running→complete|failed` status transitions. `rederiveRun` is a plain public `action` (fetch → validate → audit → return) because it writes no status and is a one-shot proof. **`generateRecommendations` follows `rederiveRun`** — a direct action: interpretation does not (in 5.3) drive a run-status machine, and the bounded redraft loop lives server-side in the engine (one HTTP call). This keeps 5.3 tractable. The tradeoff: a long interpretation (≤10-min bound, NFR-7) could approach Convex action time limits with large triangles; the async `202 + callback` upgrade (already headroom in AD-7, deferred from 4.2) is where that goes if it bites — flagged in deferred-work. The interpretation *status/subscription* surface 5.5 needs (skeleton table, "redrafting attempt N of M", failure state) is 5.5's to build on top; 5.3 delivers the persist+audit substrate.
