# Story 6.2: Submit for Review with Draft Lock

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an Analyst,
I want to submit a draft for review and assign a Senior Actuary,
so that the hand-off moment is explicit and race-free. (FR-13, UX-DR13, UX-DR14)

## Acceptance Criteria

1. **Given** a draft report, **When** the Analyst submits for review via an explicit dialog restating what will be recorded, **Then** status becomes `awaiting_review`, the draft locks read-only for the Analyst, the assignment appears in the assignee's review queue, and the submission is audit-logged (UX-DR14), **And** the Analyst's approval bar shows "Awaiting Senior Actuary review" with the assign control (UX-DR13).
2. **Given** the submission action, **When** the server processes it, **Then** no optimistic UI is used — the state flips only on server acknowledgment, **And** convex-test verifies the lock (Analyst edit attempts rejected server-side) and queue visibility rules.

## Scope decisions baked into this story (read before coding)

These are deliberate calls the dev agent must honor, not open questions. Rationale in Dev Notes §D1–D9. The genuinely-open questions for Rohan are at the very end (§Questions for Rohan) — they do **not** block starting.

- **D1 — Submit is a member mutation `submitReportForReview`; the status flip is a single Convex transaction with a `status === "draft"` precondition — that IS "race-free".** `requireMember(ctx, workspaceId)` is the FIRST statement (AD-4). It loads the `by_run` report row, guards existence + tenancy (`REPORT_NOT_FOUND`, no existence leak — mirrors `editReserveReport`, `runs.ts:1488`), guards `status === "draft"` else `REPORT_NOT_SUBMITTABLE`, then patches `status: "awaiting_review"`, `submittedBy`, `submittedAt`, `assignee`. Two concurrent submits: Convex mutations are serializable transactions, so one wins and the second sees `status: "awaiting_review"` → `REPORT_NOT_SUBMITTABLE`. "Race-free" is transactional serializability + the draft-status precondition — NOT client coordination. Audits `report.submittedForReview` with `{ runId, contentVersion, assignee }` (lean; no section text/figures — AD-1/AD-6). `returns`: `v.null()`. **Not `requireRole`** — submitting a draft is a member action (the *approve* is the Senior-Actuary-gated action, 6.4). (See §D1.)
- **D2 — The draft lock is ALREADY enforced by 6.1's `editReserveReport` guard; 6.2 only flips the status that makes it bite.** `editReserveReport` (`runs.ts:1497-1503`) already throws `REPORT_NOT_EDITABLE` on any non-`draft` row, and `ReportEditorView` already renders read-only when `report.status !== "draft"` (`ReportTab.tsx:285`). So the "draft locks read-only, Analyst edit attempts rejected server-side" behavior is a **consequence** of the D1 status flip — 6.2 adds NO new edit-guard, it just makes `awaiting_review` reachable and proves the lock with a convex-test (submit → edit attempt → `REPORT_NOT_EDITABLE`). The lock is server-side (the mutation guard), never UI-only (AC-2). (See §D2.)
- **D3 — Schema: add `assignee` / `submittedBy` / `submittedAt` as OPTIONAL product-plane columns on `reserveReports`.** OPTIONAL (unlike 6.1's required content-version columns) because every existing draft row (machine-drafted or manual) legitimately lacks them until submission — a `draft` has no submitter/assignee. No engine-contract change (`reserveReportValidator` / `convex/lib/engineContract.ts` untouched — these are product-plane columns, NOT part of the drift-checked `report` document, AD-10; confirm `tests/engine-contract.test.ts` stays green). Reuse the existing `by_workspace` index for the review-queue query (D5) — no new index needed. (See §D3.)
- **D4 — The assignee is advisory routing metadata, recorded but role-UNverified server-side; the member picker is client-side via Clerk `useOrganization`.** Convex has NO Clerk-backend seam (roles live only in the JWT, never in tables — AD-4; there is no `clerkClient` in `convex/`), so the server cannot verify the chosen assignee actually holds `senior_actuary`. The security-critical parts (the lock, the status flip, the audit entry) are FULLY server-enforced; the `assignee` id is soft routing only — the 7.3 review queue is Workspace-scoped by `awaiting_review` status (any Senior Actuary can approve in 6.4, and the 7.3 dashboard queue shows *submitter + date*, epics.md:812), so a mis-typed assignee never gates approval. The picker lists org members with `org:senior_actuary` via the Clerk client hook `useOrganization({ memberships: true })` (zero new dependency; `@clerk/nextjs` already ships it). If the org has ZERO senior actuaries, submission is still allowed with `assignee` unset (edge case, documented). Do NOT add a Clerk-backend integration to Convex for this story. (See §D4.)
- **D5 — The review-queue DATA PRIMITIVE `listReportsAwaitingReview` ships now; the DASHBOARD surface is Epic 7.3.** A Workspace-scoped query (`requireMember` first, `by_workspace` index, filter `status === "awaiting_review"`) returning per row `{ reportId, runId, submittedBy, submittedAt, assignee }` (NO figures/section text — AD-1 lean). AC-1's "the assignment appears in the assignee's review queue" is made TRUE and testable by this query now (convex-test: an `awaiting_review` report appears; a `draft` does not; cross-tenant rows are invisible). The dashboard rendering ("recent Runs … reports awaiting review … submitter and date", epics.md:805-812, UX-DR18) is Story 7.3 — 6.2 builds the primitive, 7.3 renders it. Do NOT build a dashboard page here. (See §D5.)
- **D6 — The submit confirmation dialog is the FIRST audit-generating dialog (UX-DR14); introduce `components/ui/dialog.tsx` (radix `Dialog`, ZERO new dependency).** `radix-ui` (already a dep, `package.json:20`) exports `Dialog` — the same unified-package idiom `components/ui/tooltip.tsx` + `popover.tsx` use. Radix gives focus-trap, `Esc`-to-dismiss, initial-focus control, and `aria-modal` for free — never hand-roll modal behavior. The dialog restates what will be recorded ("This submits Draft v{n} for review and assigns {assignee}. Your submission will be logged. The draft will lock read-only while it is under review.", UX-DR14). Initial focus on **Cancel** (UX-DR14 safety posture for audit-generating dialogs — matches the approval dialog 6.4 will build on the same primitive). NO optimistic UI — the confirm button calls the mutation and the state flips on server ack only (AC-2; `useState` pending flag, mirroring `ReportEditorView`'s save). 6.3 (override) and 6.4 (approve/publish) REUSE this `Dialog`. (See §D6.)
- **D7 — The sticky approval bar `ReportApprovalBar` is introduced here, ANALYST-side only; the Senior-Actuary Approve controls are 6.4.** New `components/report/ReportApprovalBar.tsx`, rendered by `ReportTab` below the editor when a report exists (UX-DR13, EXPERIENCE.md:74, mockup `report-review.html`). States: **`draft`** → the assign control (Senior-Actuary picker) + a "Submit for review" primary opening the D6 dialog; **`awaiting_review`** → "Awaiting Senior Actuary review", the assignee name, and (read-only, the editor is already locked). The bar's Approve & Publish region, the "N claims · N citations resolve" count, and the diff-since-draft link are **6.4** (UX-DR13 second half) — 6.2 builds the shell + the analyst controls and leaves a clearly-marked seam for 6.4. Do NOT build the Approve action or the resolution count here. The `awaiting review` StatusBadge already exists (`StatusBadge.tsx:34`) — surface it, don't rebuild it (D9). (See §D7.)
- **D8 — Current-user role gating on the client is Clerk `useAuth().orgRole` (normalized); the server guard stays `requireMember`.** The submit control shows for a `draft` to any member (submit is a member action — D1). The role split (Analyst submits/assigns vs Senior Actuary approves) is a 6.4 concern for the Approve control; 6.2's bar shows submit+assign on a `draft` and the awaiting-review state otherwise, for everyone. Server authority is `requireMember` in the mutation, never the client role (AD-4 — UI-hiding is never sufficient; the `status`-precondition guard is the real enforcement). Reuse the `normalizeRole` semantics from `convex/lib/guards.ts:24` conceptually (Clerk emits `org:senior_actuary`); the client only needs it to filter the assignee picker (D4), not to authorize. (See §D8.)
- **D9 — StatusBadge already carries `awaiting review`; surface it, and extend the report sub-line with "· submitted by {name}".** No StatusBadge change (`StatusBadge.tsx:18,34` already maps `"awaiting review"` → caution). The editor sub-line (`ReportTab.tsx:325-327`, "Draft v{n} · {provenance}") gains "· submitted by {submittedBy}" when `status === "awaiting_review"` (mockup `report-review.html:56`). Keep it display-only string formatting (AD-1). (See §D9.)

- **AD-4 — no anonymous access; the enforced lock is server-side.** `submitReportForReview` + `listReportsAwaitingReview` call `requireMember` FIRST (both enumerated in `authGuard.test.ts`, Task 3). - **AD-6 / FR-15 — every submission is audit-logged.** `report.submittedForReview` (actor + `contentVersion` + `assignee`) via the sole writer `appendAuditEntryInTransaction`; the chain stays append-only + hash-chained. - **AD-5 — the gate governs machine-drafted content only; submission is a human-owned lifecycle event, never re-gated.** `submitReportForReview` does NOT call the Provenance Gate / any engine endpoint — it is a pure product-plane status transition (AD-5: human-owned actions are audit-logged, not gated). ### §D1 — submit as a race-free single-transaction status flip

"Race-free" (the AC's word) is achieved by Convex's transactional serializability + a `status === "draft"` precondition, NOT by client coordination or optimistic locking. Two Analysts (or an Analyst + a stale tab) submitting the same draft concurrently: Convex serializes the two mutations; the first patches `draft → awaiting_review`; the second re-reads `awaiting_review` inside its own transaction and throws `REPORT_NOT_SUBMITTABLE`. There is no window where both succeed, and no lost update. The same precondition makes a double-click idempotent (the second click's mutation fails cleanly; the UI shows the readable error, keeps the dialog open). This is the standard Convex "guard-on-current-state" pattern, the same shape as `editReserveReport`'s `REPORT_NOT_EDITABLE`.

### §D2 — the lock is a consequence, not new code

6.1 deliberately built `editReserveReport`'s `REPORT_NOT_EDITABLE` guard (and `ReportEditorView`'s read-only path) BEFORE `awaiting_review` was reachable, precisely so 6.2's lock would be "real and testable now" (6.1 story D5, `schema.ts:186-189`). 6.2 therefore adds NO edit-guard: flipping `status` to `awaiting_review` makes the existing server guard reject edits and the existing client `editable = status === "draft"` render read-only. 6.2's job is (a) the status flip (Task 2.1), (b) proving the lock with a convex-test (submit → edit → `REPORT_NOT_EDITABLE`, Task 8.1b), and (c) the analyst-facing bar that shows the awaiting-review state. Do NOT reimplement the lock in the bar or the tab — the single source of truth is the mutation guard + the `status` field.

### §D4 — the assignee is advisory; the picker is client-side Clerk

The AC says "assign a Senior Actuary", but Convex cannot verify a chosen user's org role (roles live only in the Clerk JWT of the *acting* user — AD-4; there is no `clerkClient` in `convex/`, and adding one is a new integration seam out of scope for a lifecycle story). So `assignee` is stored as advisory routing metadata (a Clerk user id string), unverified. This is safe because: (a) the 7.3 dashboard review queue is Workspace-scoped by `awaiting_review` status and shows *submitter + date* (epics.md:812) — it does not gate on assignee; (b) any Senior Actuary can approve in 6.4 (`requireRole`); (c) a mis-assigned report is a routing hint, not a security boundary. The assignee PICKER is populated client-side from Clerk `useOrganization({ memberships: … })`, filtering members whose role normalizes to `senior_actuary` (Clerk emits `org:senior_actuary`) — zero new dependency, no Convex change. Empty-list edge: submit allowed with `assignee = null`. If strict server-side assignee validation is ever required, it needs a Clerk-backend action in Convex — flagged in deferred-work §6.2.
