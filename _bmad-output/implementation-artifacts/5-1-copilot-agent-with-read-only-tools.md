# Story 5.1: copilot_agent with Read-Only Tools

Status: review

## Story

As the product,
I want an Agno-hosted Gemini agent whose only data access is read-only typed views over the current Run,
so that the interpretation layer is structurally incapable of touching anything else. (FR-9)

## Acceptance Criteria

1. **Given** `copilot_agent`, **When** it is constructed for an interpretation request, **Then** it uses Agno's model abstraction over the official `google-genai` SDK with model ID `gemini-3.1-flash-lite` read from engine_service config — never raw REST (AD-8, thought-signature handling).
2. **And** its tool surface exposes only read-only, provider-neutral JSON Schema views over the request's in-memory ResultSet and DiagnosticsBundle (list diagnostics, get diagnostic by ID, get result fields, get run metadata) — no filesystem, network, Convex, or write operations (FR-9).
3. **Given** an agent session, **When** it completes, **Then** the full transcript (prompt, every tool call and result, response) is captured and returned to the caller for audit logging — no durable state remains in Agno sessions (AD-3).
4. **And** pytest verifies tool read-onlyness (no mutating call exists) and transcript completeness on a stubbed model.

## Dev Notes

### Architecture compliance (non-negotiable)

- **AD-8 — read-only views over validated output (the whole point of this story).** The agent's data-access boundary is exactly the four tools in Task 3, each a closure over the request's in-memory `ResultSet`/`DiagnosticsBundle`. No filesystem, no network, no Convex, no writes. Tool schemas + prompts stay **provider-neutral** (plain JSON Schema derived from Python type hints); the single provider-specific seam is `build_gemini_model` naming `agno.models.google.Gemini`. The model is reached ONLY through Agno over the official `google-genai` SDK (which handles Gemini 3.x thought signatures in the tool loop) — never raw REST.
- **AD-3 — Convex is the sole system of record; agent state is transient.** No `db`/session store on the `Agent`; the interpretation transcript is the only artifact that leaves, returned to the caller for audit logging. Nothing durable in Agno.
- **AD-2 / layering — `copilot_agent` is a shell above the pure core.** It imports `reserving_engine` (public API) + `agno` + stdlib; it must NOT import `engine_service` (upward) or Convex/Clerk (cross-plane). The import-linter layer reorder (Task 1) is what makes engine_service→copilot_agent legal while keeping copilot_agent→reserving_engine the only downward dep. `reserving_engine` purity (forbidden-modules contract) is untouched.
- **AD-1 — numbers originate only in the engine.** The tools READ figures from the ResultSet/DiagnosticsBundle verbatim and compute nothing (no sums, deltas, ratios). Any arithmetic the model appears to "need" is already a field on the engine models (that was the 2.2/2.4 design intent). If a value seems missing, it is a real engine gap — do not compute it in a tool.
- **AD-10 — the wire is the camelCase contract.** Tools emit `model_dump(mode="json", by_alias=True)` and the transcript models reuse `_MODEL_CONFIG`, so everything the model and the Audit Log see speaks the same `runId`/`schemaVersion`/`triangleHash`/`ldfStability` shapes as the rest of the system. Diagnostic IDs are resolved by `resolve_diagnostic` (dict lookup), NEVER by splitting the `dx:` string.

### The layering gotcha (read this first)

The scaffolded `pyproject.toml` has `layers = ["engine_service | copilot_agent", "reserving_engine"]`. The `|` declares the two shells as **independent peers**, which import-linter enforces as "these two may not import each other." That directly contradicts the architecture, where engine_service **hosts** copilot_agent (spine diagram `SVC --> AGENT`; AD-8 "copilot_agent hosting"). CI runs `uv run lint-imports` and will fail the moment engine_service imports copilot_agent. Task 1 fixes this by reordering to three genuine layers `["engine_service", "copilot_agent", "reserving_engine"]` (downward-only). This is a legitimate contract correction, not a workaround — it makes the linter match the intended dependency graph. Do it first so the module you build lands green.

### Agno 2.7.3 API — the exact surface (verified against the installed package)

- Imports: `from agno.agent import Agent`, `from agno.models.google import Gemini`. (Confirmed importable in `engine/.venv`.)
- `Gemini(id: str = "gemini-3.5-flash", api_key: Optional[str] = None, ...)`. **`api_key` defaults to reading `GOOGLE_API_KEY` from env** (`agno/models/google/gemini.py:175`). Our config var is `GEMINI_API_KEY`, so **pass `api_key=` explicitly** — do not rely on Agno's env default (it reads the wrong name).
- `Agent(model=..., tools=[callable, ...], instructions=..., telemetry=False, markdown=False)`. `tools` accepts a list of plain callables (Agno derives the JSON Schema from each signature + docstring), `Toolkit`s, `Function`s, or dicts — use **plain typed callables** for provider-neutral schemas. `telemetry` defaults to **True** (external analytics egress) → set **False**.
- `agent.run(prompt)` returns a `RunOutput` (dataclass) with `.content` (final text), `.messages: list[Message]`, and `.tools: list[ToolExecution]`. Build the transcript from `.messages`.
- `agno.models.message.Message` fields include `role`, `content`, `tool_calls`, `tool_name`, `tool_args`, `tool_call_id`, and a `to_dict()` method — everything the audit transcript needs. (`ToolExecution` is not a Pydantic model; prefer `.messages` for the transcript.)
- Stubbed model: subclass `agno.models.base.Model` (ABC) — the abstract methods are `invoke`, `ainvoke`, `invoke_stream`, `ainvoke_stream` (plus provider-parse hooks around line 567/580 of `agno/models/base.py`). `ModelResponse` (`agno/models/response.py`) carries `content`, `tool_calls: list[dict]`, `role`. A minimal fake that returns a canned tool-call `ModelResponse` then a canned final `ModelResponse` exercises the real tool loop. If this proves brittle on 2.7.3, use the seam-level assertions in Task 7.6's fallback and rely on the pure `build_transcript` test (7.5) for the transcript contract.

### The engine contract these tools wrap (source-of-truth field names)

From `reserving_engine` (all frozen, camelCase on the wire):
- `ResultSet(schema_version, lineage: Lineage, method_results: tuple[MethodResult, ...])`.
- `Lineage(engine_version, chainladder_version, triangle_hash, parameters: RunParameters)`; `RunParameters(methods: tuple[Literal["chain_ladder","bornhuetter_ferguson","mack"], ...], apriori_loss_ratios: tuple[AprioriLossRatio, ...])`.
- `MethodResult(method, development_factors: tuple[DevelopmentFactor,...], origin_results: tuple[OriginResult,...], total_mack_std_err)`; `OriginResult(origin, ultimate, ibnr, mack_std_err?, reserve_low?, reserve_high?)`; `DevelopmentFactor(from_dev, to_dev, factor)`.
- `DiagnosticsBundle(schema_version, run_id, triangle_hash, ldf_stability, ave, cl_bf_divergence: tuple|None, residuals)`. `cl_bf_divergence is None` ⇒ CL+BF did not both run (absent ≠ empty).
- Diagnostic elements each carry `id` (`dx:{runId}:{kind}:{key}`): `LdfStabilityElement`, `AveElement`, `ClBfDivergenceElement`, `ResidualElement`. Resolve with `resolve_diagnostic(bundle, id)` → raises `UnknownDiagnosticIdError` on miss. Iterate all elements with the same order `diagnostics._iter_elements` uses.

### Scope boundary — what is NOT in 5.1

- **The Provenance Gate** (placeholder rendering + numeric-token checker) is **Story 5.2** — this story does NOT render `{{rs:…}}`/`{{dx:…}}` or check numbers.
