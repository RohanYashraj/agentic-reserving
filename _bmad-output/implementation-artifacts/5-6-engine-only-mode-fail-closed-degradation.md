# Story 5.6: Engine-Only Mode Fail-Closed Degradation

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an Analyst,
I want the app to degrade cleanly when the model is unavailable or the cost ceiling is hit,
so that an AI outage never blocks the quarter. (FR-12, NFR-2)

## Acceptance Criteria

1. **Given** persistent model-API failure or a per-Run token/cost ceiling breach (ceiling values from `engine_service` config), **When** an Interpretation attempt fails, **Then** the Interpretation is marked failed cleanly on the run record, Engine-Only Mode is derived server-side (never a client-side guess), and the mode transition is audit-logged (AD-9, FR-12).
2. **Given** Engine-Only Mode, **When** any surface renders, **Then** the full-bleed caution banner shows "Engine-Only Mode ‚Äî interpretation unavailable" with a "what still works" link, non-dismissable while the condition holds, announced `aria-live="assertive"` once with entry/exit toasts once (UX-DR4).
3. **And** Interpretation actions are disabled with tooltips while upload ‚Üí Run ‚Üí Diagnostics remain fully functional (verified by test) (NFR-2).
4. **And** exiting the condition restores Interpretation and audit-logs the exit.

> **This is a FULL-STACK story spanning all three planes** ‚Äî engine (cost/timeout ceiling enforcement + a health probe), Convex (server-derived mode state + audit + the two actions' fail-closed wiring), and frontend (the global banner + disabled actions). It is the largest Epic-5 story and the epic's **hardening** step (build-sequence: "agent layer last ‚Üí hardening"). Tasks are ordered engine ‚Üí contract ‚Üí Convex ‚Üí frontend ‚Üí tests and each is independently testable. The `engine.model_unavailable` signal already exists end-to-end (5.3/5.4) and every comment that says "Story 5.6 keys Engine-Only Mode on it" points here ‚Äî **do NOT invent a new model-outage signal; consume the existing one.** See Dev Notes ¬ßWhat already exists.

## Scope decisions baked into this story (read before coding)

These are deliberate calls the dev agent must honor, not open questions. Rationale in Dev Notes ¬ßD1‚ÄìD7.

- **D1 ‚Äî Two triggers, split scope.** Model-plane unavailability (`engine.model_unavailable`) drives a **workspace-global** Engine-Only Mode (the banner). A per-Run **token/cost ceiling breach** and the **interpretation timeout** are **per-Run** clean failures recorded on that run's record + audit-logged, surfaced on that run's Interpretation tab ‚Äî they do **NOT** flip the global banner. Both triggers satisfy AC-1's "marked failed cleanly on the run record + mode transition audit-logged"; only model-availability derives the global mode. (See ¬ßD1 for why + the rejected "both-global" alternative.)
- **D2 ‚Äî The mode is persisted-and-derived server-side, never client-guessed.** A per-Workspace `interpretationModes` row is the durable system-of-record for Engine-Only Mode, written **only** server-side from the engine's typed response inside the Convex actions. Banner + tab **subscribe** to `getInterpretationMode` (reactive, survives reload, no polling ‚Äî FR-20). "Derived server-side" = the derivation lives in the Convex action; the client only reads the result. (See ¬ßD2.)
- **D3 ‚Äî Entry on the first failed attempt; recovery via a server probe.** Mode ENTERS when a `generateRecommendations`/`generateReserveReport` attempt returns `engine.model_unavailable`. Because the trigger button is disabled in the mode (AC-3), recovery is the banner's **"Retry" / "Check again"** ‚Üí a `probeInterpretationMode` action ‚Üí the new engine `GET /interpretation/health` (builds the model; 200 ok / 503 `model_unavailable`) ‚Üí clears the mode + audit-logs exit on success. No polling; user-initiated. A proactive on-load probe (mode visible before ANY attempt) is deferred. (See ¬ßD3.)
- **D4 ‚Äî Transitions are edge-triggered + idempotent; "once" is enforced at the edge.** The internal `transitionEngineOnlyMode` mutation writes the row **and** audit-logs (`mode.engineOnlyEntered` / `mode.engineOnlyExited`) **only when the boolean actually flips** ‚Äî re-entering while already Engine-Only is a no-op (no duplicate audit). The banner fires the `aria-live="assertive"` announcement and the entry/exit toast **once per edge** (a `useRef` comparing previous vs current). (See ¬ßD4.)
- **D5 ‚Äî Ceiling/timeout/attempts become `engine_service` config; enforcement is in-flow with an injected clock.** `INTERPRETATION_MAX_ATTEMPTS` (default 3, replaces the `MAX_ATTEMPTS` module constant), `INTERPRETATION_TOKEN_CEILING` (int, `None` = unbounded), `INTERPRETATION_TIMEOUT_SECONDS` (default 600 = NFR-7's ‚â§10 min). The redraft loop accumulates per-attempt token usage and checks the budget + an **injected monotonic deadline** between attempts; a breach raises typed `CostCeilingExceededError` / `InterpretationTimeoutError` ‚Üí 503 envelope codes `cost_ceiling_exceeded` / `interpretation_timeout` (fail-closed, like `model_unavailable`). The clock/deadline is **injected** (not `time.monotonic()` inline) so the flow stays deterministic under tests (mirrors the `build_model` seam; preserves the module's "no clock" property). (See ¬ßD5.)
- **D6 ‚Äî The manual report template shell is Epic 6, not 5.6.** FR-12's "manual report template shell" is explicitly completed in Epic 6 (`epics.md:113` + Story 6.1). 5.6 delivers: server-side mode derivation, the global banner, disabled interpretation actions with tooltips, and the durable per-Run interpretation-failure state. The Report tab / manual template lands in Epic 6.1. 5.6's job is to keep upload ‚Üí Run ‚Üí Diagnostics fully functional (the NFR-2 invariant, verified by test) ‚Äî NOT to build the manual drafting surface. (See ¬ßD6.)
- **D7 ‚Äî Tooltip from the already-present `radix-ui` unified package (zero new dep); toast hand-rolled minimal.** `components/ui/tooltip.tsx` wraps `Tooltip` from `radix-ui` ‚Äî the SAME idiom `components/ui/popover.tsx` uses (`import { Popover as PopoverPrimitive } from "radix-ui"`), so **no new dependency**. No toast library exists in the repo (no sonner/shadcn-toast); the entry/exit toast is a minimal, single-instance, auto-dismissing surface owned by `EngineOnlyBanner`, fired on the mode-transition edge. Document the choice; do NOT add a toast dep silently. (See ¬ßD7.)

### ¬ßD1 ‚Äî two triggers, split scope (model-outage ‚Üí global; cost/timeout ‚Üí per-Run)

AC-1 lists two triggers ‚Äî "persistent model-API failure **or** a per-Run token/cost ceiling breach." They have different natural scopes:

- **Model unavailability is environmental/global** ‚Äî the model plane is down, so interpretation is unavailable for the WHOLE Workspace. The PRD glossary defines the mode this way: "Engine-Only Mode ‚Äî degraded operating mode **when the interpretation model is unavailable**" (`prd.md:60`). This drives the global banner (EXPERIENCE.md:88 "Global"). It self-heals: when the model returns, the next reachable call (or the banner's probe) clears it.
- **A per-Run cost/token ceiling breach (or the ‚â§10-min timeout) is per-Run** ‚Äî the model IS reachable; THIS Run exhausted its budget. Flipping the whole Workspace into Engine-Only Mode on one expensive Run would (a) be semantically wrong ("interpretation unavailable" when the model is fine), and (b) **deadlock recovery** ‚Äî the global mode disables the trigger, but a cost-breach can't "un-breach" itself, so nothing would clear it. So a cost/timeout breach fails THAT Run's interpretation cleanly (recorded on the run record + `run.interpretationFailed` audit), surfaced on that run's tab, and the user can re-run interpretation (the model is up).

Both triggers satisfy AC-1's literal requirements ‚Äî "marked failed cleanly on the run record" (both write `runs.interpretationFailure` + `run.interpretationFailed`) and "the mode transition is audit-logged" (the model-outage case logs `mode.engineOnlyEntered`; the per-Run failure IS the audited transition-to-failed for that Run). **Rejected alternative:** treating cost-breach as ALSO global ‚Äî rejected for the deadlock + semantic reasons above. Flagged in deferred-work so a reviewer sees the call was deliberate.

### ¬ßD2 ‚Äî persisted-and-derived, not client-guessed; the durable field 5.5 deferred

"Derived server-side, never a client-side guess" (AD-9) + "survive reload" + "any surface renders" (AC-2) together require a **persisted, server-written, client-subscribed** representation ‚Äî a pure on-read derivation can't be reactive (Convex queries can't call the engine; only actions can). So: `interpretationModes` (per-Workspace) is written ONLY by server code (the actions + the probe), from the engine's typed response; the banner + tab subscribe via `getInterpretationMode` (`useQuery`, live, no polling ‚Äî FR-20). This also closes Story 5.5's deferred line 205 ("a durable interpretation-status field so the running state survives reload"): 5.6 adds the durable **failed** state (`runs.interpretationFailure`) ‚Äî the durable *running* state still needs the async transport and stays deferred.

### ¬ßD3 ‚Äî entry on first failure; recovery via a server probe

The mode ENTERS the first time a real interpretation attempt returns `engine.model_unavailable` (the actions catch it, Task 5.5/5.6). It does NOT proactively trip before any attempt (a fresh page load with the model already down shows the mode only after the first failed attempt OR an explicit probe) ‚Äî a proactive on-load derivation is deferred (would need every surface to fire a probe on mount, which flirts with polling). Because AC-3 disables the trigger button in the mode, recovery is the banner's explicit **"Retry"/"Check again"** ‚Üí `probeInterpretationMode` ‚Üí `GET /interpretation/health` (a cheap model-build, NOT a full interpretation) ‚Üí clears the mode on 200 (audit `mode.engineOnlyExited` + exit toast). User-initiated, no polling. The success path of the two interpretation actions ALSO clears the mode (self-heal on the next reachable call), so a user who re-runs on a different Run after the model returns exits the mode without touching the banner.

### ¬ßD4 ‚Äî edge-triggered + idempotent transitions ("once")

`transitionEngineOnlyMode` writes + audits ONLY on an actual boolean flip (compare the stored `engineOnly` to the requested value; equal ‚Üí `{changed:false}`, no write, no audit). This makes "entry/exit toasts once" and "aria-live assertive once" (UX-DR4) enforceable at the source: a second `model_unavailable` while already Engine-Only is a no-op, so no duplicate `mode.engineOnlyEntered`. The banner independently guards its announcement/toast with a `useRef` on the previous value so it fires only on the observed edge (and NOT on initial mount, where `prev.current === undefined`). Two layers of "once": the server (no duplicate audit) and the client (no duplicate announcement).

### ¬ßD5 ‚Äî ceiling/timeout/attempts as config; injected clock keeps the flow deterministic

`recommendations_flow.py`/`report_flow.py` guarantee determinism under a scripted model ("no clock, no randomness, no logging" ‚Äî `recommendations_flow.py:16-20`). A wall-clock timeout would break that. So the deadline is derived from an **injected** `now: Callable[[], float]` (default `time.monotonic`) ‚Äî tests pass a fake `now` and the flow stays deterministic; prod passes the real monotonic clock (allowed in the shell, AD-2). The token ceiling accumulates `InterpretationResult.token_count` across attempts (Task 2.1 ‚Äî verify agno's usage accessor via context7). Both are checked between attempts; a breach raises a typed fail-closed exception ‚Üí 503 envelope (NOT a redraftable rejection, NOT a 500 bug). `INTERPRETATION_MAX_ATTEMPTS` replaces the `MAX_ATTEMPTS` constant as the "calls" bound (PRD ¬ß8: "config ceiling on tokens/calls").
