# Story 6.4: Approve and Publish

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a Senior Actuary,
I want to approve and publish a report through an explicit, fully-logged sign-off,
so that the approval moment carries the weight my signature does. (FR-13, UX-DR13, UX-DR14)

## Acceptance Criteria

1. **Given** a report awaiting review, **When** the approval bar renders for a Senior Actuary, **Then** it shows the citation-resolution count ("41 claims · 41 citations resolve"), a diff-since-draft link, and Approve & Publish — disabled with the failing sentence linked if any citation fails to resolve (UX-DR13), **And** the approval dialog restates version, citation count, any overrides, and "This will be logged and the published version cannot be edited", keyboard-operable with focus trapped and initial focus on Cancel (UX-DR14).
2. **Given** confirmation, **When** publish executes, **Then** approver identity, timestamp, and the exact approved content version are audit-logged, the report becomes immutable with the `published` badge, and post-publication changes require "Start new version" creating a new draft superseding record (FR-13), **And** confirmation is on server ack only, and convex-test verifies: Analyst publish rejection, immutability of published content, unresolved-citation blocking, and the approval audit entry.

## Scope decisions baked into this story (read before coding)

These are deliberate calls the dev agent must honor, not open questions. Rationale in Dev Notes §D1–D9. The genuinely-open questions for Rohan are at the very end (§Questions for Rohan) — they do **not** block starting.

- **D1 — `approveAndPublish` is the report-lifecycle sibling of 6.2's `submitReportForReview`, but Senior-Actuary-gated: `requireRole(ctx, workspaceId, "senior_actuary")` is the FIRST statement, and the precondition is `status === "awaiting_review"`.** It follows the exact established transition shape (guard → `by_run` unique lookup → `REPORT_NOT_FOUND` on null/cross-tenant → status-precondition throw → `ctx.db.patch` → `appendAuditEntryInTransaction`, `runs.ts:1837-1892`). Precondition: `row.status !== "awaiting_review"` → `ConvexError { code: "REPORT_NOT_APPROVABLE", message: "This Reserve Report is not awaiting review and cannot be approved." }` (a `draft` not yet submitted, or an already-`published` row, is rejected — this precondition + Convex OCC serializability is the race guard, mirroring 6.2's `REPORT_NOT_SUBMITTABLE`, §D1). This is the SECOND production caller of `requireRole` (after 6.3's `overrideRecommendation`, `runs.ts:618/633`) — an Analyst's publish attempt is rejected SERVER-SIDE with `FORBIDDEN`, satisfying AC-2's "Analyst publish rejection" (never UI-hiding — AD-4; D7's client `canApprove` is display courtesy only). (See §D1.)
- **D2 — The approval BLOCKER (server reject + client disable) is "any uncited claim", computed via 6.1's `uncitedSentences` — the single source of truth the helper was BUILT for.** `convex/lib/citationMarker.ts:193-203` `uncitedSentences(text)` returns figure-bearing sentences carrying no `[[cite:...]]` marker; its docstring (`:16-17`) explicitly earmarks it as "6.4's approval blocker". `approveAndPublish` runs it over the four section texts; if ANY section has an uncited sentence, it throws `ConvexError { code: "REPORT_HAS_UNCITED_CLAIMS", details: { sections: [<sectionKey>...] } }` and does NOT publish (AC-2 "unresolved-citation blocking"). The server needs NO DiagnosticsBundle for this — `uncitedSentences` is a pure marker/figure string check, server-available in `convex/lib`. This makes the blocker identical on both planes (the client disables Approve using the same helper via the D6 client resolver, the server enforces it) and reuses 6.1's tested code verbatim — no reimplementation. (See §D2/§D6.)
- **D3 — Published immutability = the existing edit/submit guards + a NEW append-only `publishedReportVersions` snapshot table; NOT a `by_run.unique()`-breaking multi-row supersession.** Every report function reads the row via `.withIndex("by_run").unique()` (7 call sites). Adding a second `reserveReports` row per Run (the literal reading of "new draft superseding record") would break `.unique()` across all of them — a broad, risky refactor. Instead: `approveAndPublish` **appends** an immutable snapshot `{ workspaceId, runId, reportId, contentVersion, report (frozen copy), approvedBy, approvedAt, overrideCount }` to a new append-only table `publishedReportVersions` (never patched/deleted — the AD-6 append-only ethos, mirroring 6.3's separate `recommendationOverrides` precedent, `schema.ts` 6.3 block), AND flips the working `reserveReports` row `status → "published"`. The working row is edit-locked by the EXISTING `editReserveReport` (`REPORT_NOT_EDITABLE`, `runs.ts:1691`) and `submitReportForReview` (`REPORT_NOT_SUBMITTABLE`) guards — a `published` row is already immutable to those. The signed content is preserved forever in the append-only snapshot. AC-2 "immutability of published content" is thus doubly guaranteed (the working row rejects edits/submits/re-publish; the snapshot is append-only). The ERD's `REPORT ||--o{ REPORT : "new version supersedes"` is realized as the working row's `supersedes` link → the prior published snapshot (D5), NOT a same-table row clone. (See §D3.)
- **D4 — The approval records `approvedBy`/`approvedAt` on the working row (for the inline "Approved by … · Logged" record, UX-DR19) AND in the append-only snapshot; the audit is ONE lean event `report.published`.** Add OPTIONAL `approvedBy: v.optional(v.string())` / `approvedAt: v.optional(v.string())` columns to `reserveReports` (optional — every pre-publish row lacks them). `approveAndPublish` patches them (`approvedBy: actor`, `approvedAt: new Date(Date.now()).toISOString()` — repo clock convention, `runs.ts:1873`; never a clock in the engine) so the published composition renders the inline record without a snapshot round-trip. The Audit Log entry is `eventType: "report.published"`, `payload: { runId, contentVersion, overrideCount }` (lean — no section text/figures, matching every `report.*` audit; AD-1/AD-6). Approver identity is the entry's `actor`; the timestamp is the entry's own hash-chained `timestamp` — together they ARE AC-2's "approver identity, timestamp, and the exact approved content version". ONE event, not `report.approved` + `report.published` (Approve & Publish is a single atomic action; AD-6 lists "approval" as one category). (See §D4.)
- **D5 — "Start new version" is `startNewReportVersion`, a `requireMember` mutation gated on `status === "published"`; it re-opens the SAME working row to `draft` (contentVersion + 1), linking `supersedes` → the last published snapshot.** Post-publication editing is a member action (an Analyst starts the revision; re-publishing STILL requires `requireRole` approval — no privilege escalation, so `requireMember` is correct, matching `editReserveReport`). Precondition `status !== "published"` → `REPORT_NOT_PUBLISHED`. It loads the latest `publishedReportVersions` row for the Run (`by_run`, max `contentVersion`), then patches the working row: `status: "draft"`, `contentVersion: row.contentVersion + 1`, `updatedBy: actor`, `updatedAt: now`, `supersedes: <snapshotId>`, and clears `approvedBy`/`approvedAt` (set `undefined`). The content stays as the published content, now editable again — the new draft supersedes the archived published version (which remains immutable in the snapshot table). Audit `report.newVersionStarted` with `{ runId, contentVersion, supersedes }`. This is the ONLY escape from published immutability (AC-2). Add `supersedes: v.optional(v.id("publishedReportVersions"))` to `reserveReports`. (See §D5.)
- **D6 — The client "N claims · N citations resolve" count + failing-sentence blocker: a NEW client helper `reportCitationResolution` reusing `uncitedSentences` + `resolveDiagnostic`; AD-1-clean metadata aggregation.** New `components/report/reportCitationResolution.ts`: given the four sections + the `DiagnosticsBundle`, it returns `{ totalClaims, resolvedClaims, failingSentences: { sectionKey, sentence, offset }[] }`. A "claim" is a figure-bearing sentence (via `uncitedSentences`'s figure detection). A claim FAILS if it carries no `[[cite:...]]` marker (uncited — the reachable case, 6.1 D2: humans delete but cannot forge chips) OR carries a marker whose `dxId` does not resolve via `resolveDiagnostic(buildDiagnosticIndex(bundle), dxId)` (dangling — near-unreachable through the editor, checked for display honesty). `resolvedClaims = totalClaims − failingSentences.length`; the count renders "{totalClaims} claims · {resolvedClaims} citations resolve" (UX-DR13). The bar disables Approve when `failingSentences.length > 0` and links each failing sentence to its section (reuse the section anchor / hash-scroll used by chips, `CitationChip` `window.location.hash`). This is display-only (string/metadata, no arithmetic on reserve figures — AD-1); the SERVER re-checks the uncited-claim blocker independently (D2) so the gate is never client-trusted. (See §D6.)
- **D7 — The Approve region lives in `ReportApprovalBar`'s `AwaitingReviewBar` seam (`ReportApprovalBar.tsx:224-225`); the published composition in the `published` seam (`:55`). `canApprove` mirrors 6.3's `canOverride`.** For a Senior Actuary (`canApprove === true`), the awaiting-review bar ADDS the resolution count, the diff-since-draft link, and the green **Approve & Publish** button (the ONLY green action surface, UX-DR13/DESIGN.md published-green) opening the D8 dialog; for an Analyst (`canApprove === false`) it keeps the existing "Awaiting Senior Actuary review" + assignee text unchanged (they see the state, not the action — role-gated, but the SERVER `requireRole` is the authority, AD-4). The `published` branch replaces its placeholder with: `<StatusBadge status="published" />`, the inline approval record "Approved by {approvedBy}, {date}, {time} · Logged" (UX-DR19 copy pattern, display-formatting only), a **Start new version** button (calls `onStartNewVersion`), and a `{/* 6.5: Export to Word renders here */}` seam (do NOT build export — Story 6.5). `canApprove = normalizeRole(orgRole) === "senior_actuary"` on the page, threaded page → RunDetail → ReportTab → ReportApprovalBar (the exact 6.3 `canOverride` plumbing). (See §D7.)
- **D8 — The approval dialog REUSES the 6.2 `Dialog` primitive + the `DraftBar` dialog contract exactly (UX-DR14); it restates version, citation count, and any overrides.** `DialogTitle` "Approve & publish"; `DialogDescription` restating: "This publishes Draft v{contentVersion} with {resolvedClaims}/{totalClaims} citations resolving{overrideCount ? `, including {overrideCount} recommendation override(s)` : ""}. This will be logged and the published version cannot be edited." (UX-DR14 verbatim intent + Flow-3 step 6). `onOpenAutoFocus` → initial focus on **Cancel** (`ReportApprovalBar.tsx:156-159`, the safety posture); a `useState` pending flag disables both footer buttons and swaps the confirm label ("Publishing…"); NO optimistic UI — confirm `await`s `onApprove()` and the `getReserveReport` subscription re-renders the published bar on server ack (D9); `onOpenChange` clears the inline error, Cancel/Esc NEVER fire the action (`:138-143`); a rejected publish (e.g. a race that flipped status, or a server-side uncited-claim re-check) shows the readable error inline (`role="alert"`, `text-caution`, `:171-175`) and keeps the dialog open. The overrides count comes from the existing `getRecommendationOverrides` subscription (6.3) threaded into the bar (distinct overridden origins). Zero new dependency. (See §D8.)
- **D9 — No optimistic UI, server-ack only, reload-durable; diff-since-draft compares against the machine-drafted baseline captured at first human edit.** The bar/editor derive `status`/`approvedBy` purely from the `getReserveReport` subscription; publish and start-new-version flip nothing locally — they await the mutation and let the subscription re-render (AC-2 "confirmation is on server ack only", AD-3/FR-20; a logged approval must never flicker, EXPERIENCE.md "Banned: optimistic UI for audit-generating actions"). For "diff since draft" (UX-DR13): capture the machine-drafted original into a new OPTIONAL `draftBaseline: v.optional(reserveReportValidator)` column INSIDE `editReserveReport` on the FIRST human edit (when `row.machineDrafted === true` and `row.draftBaseline` is unset — a one-conditional addition to the existing 6.1 mutation, `runs.ts:1645`), so the diff compares the current `report` against `draftBaseline` (what the interpretation layer produced). The diff link opens a read-only per-section comparison (`components/report/ReportDraftDiff.tsx`, display-only string diff — AD-1); when no baseline exists (a manual report, or a never-edited machine draft where current === original), the link renders disabled with a quiet "No analyst edits to compare" note. A rich word-level diff is deferrable (§Questions). (See §D9.)

### §D1 — approve as a race-free, role-gated, single-transaction status flip

"Race-free" is Convex transactional serializability + a `status === "awaiting_review"` precondition, NOT client coordination — the exact 6.2 pattern (`submitReportForReview`'s `REPORT_NOT_SUBMITTABLE`). Two concurrent approvals: one wins and patches `awaiting_review → published`; the second re-reads `published` inside its transaction and throws `REPORT_NOT_APPROVABLE`. A double-click is idempotent (the second fails cleanly; the dialog shows the readable error, stays open). `requireRole(senior_actuary)` FIRST makes it the second production role-gated write; the Analyst rejection (`FORBIDDEN`) is server-authoritative (the convex-test acts as `analystA` and asserts `FORBIDDEN` + no state change — AC-2), independent of the client `canApprove` (D7).

### §D2 — the blocker is `uncitedSentences`, enforced on both planes

6.1 built `uncitedSentences` in `convex/lib/citationMarker.ts` (server-available) precisely so 6.4's approval gate reuses it (its docstring says so). `approveAndPublish` runs it over the four section texts and refuses to publish (`REPORT_HAS_UNCITED_CLAIMS`) when any claim (a figure-bearing sentence) lacks a `[[cite:...]]` marker — this is AC-2's "unresolved-citation blocking", tested by seeding an `awaiting_review` report with an uncited claim and asserting the throw + no publish. The server needs NO DiagnosticsBundle (uncited-detection is a pure marker/figure check). The CLIENT disables Approve using the SAME helper via `reportCitationResolution` (D6), so the button state and the server gate never disagree; the server re-check means the gate is never client-trusted. The only failure mode NOT caught server-side is a dangling chip (a marker whose `dxId` doesn't resolve) — near-unreachable through the editor (6.1 D2: humans can delete but not forge chips; adding a chip requires a real marker), checked client-side for display honesty and flagged as a residual in deferred-work.

### §D3 — a separate append-only snapshot table, not a `by_run.unique()`-breaking second row

The AC's "new draft superseding record" reads like a second `reserveReports` row per Run, but every report function reads the row via `.withIndex("by_run").unique()` (`getReserveReport`, `editReserveReport`, `createManualReport`, `submitReportForReview`, `approveAndPublish`, `startNewReportVersion`, `storeReserveReport`) — a second row breaks `.unique()` everywhere, a broad, risky refactor. Instead, published immutability is realized as: (1) the working row flips to `published` and is edit-locked by the EXISTING `editReserveReport`/`submitReportForReview` status guards (no new guard needed); (2) an APPEND-ONLY `publishedReportVersions` snapshot preserves the exact signed content forever (never patched/deleted — the AD-6 ethos on the product plane). This mirrors 6.3's decision to put overrides in a separate append-only table rather than mutate the engine document, and keeps the `by_run.unique()` invariant intact. The ERD's `REPORT ||--o{ REPORT : "new version supersedes"` is realized as the working row's `supersedes` link → the prior published snapshot (D5) — a faithful realization (a REPORT superseding a prior published REPORT record), not a literal same-table clone. True multi-row product-plane report history (if ever needed) is flagged deferred.

### §D4 — one lean `report.published` audit + on-row approval columns

AC-2 requires "approver identity, timestamp, and the exact approved content version" audit-logged. These map exactly to the audit entry's `actor` (the `requireRole` identity's subject), its own hash-chained `timestamp`, and `payload.contentVersion`. ONE event `report.published` (Approve & Publish is atomic; AD-6 lists "approval" as a single category), lean like every other `report.*` audit (no section text/figures — the content lives on the row + the snapshot). The `approvedBy`/`approvedAt` columns on the working row let the published composition render the inline "Approved by …, … · Logged" record (UX-DR19) without a snapshot round-trip; the same values are frozen in the snapshot for the durable record. `overrideCount` (distinct overridden origins) rides in the payload so the audit reflects that the sign-off covered a report with overrides.

### §D5 — "Start new version" re-opens the working row; the published snapshot stays immutable

`startNewReportVersion` (`requireMember` — post-publication editing is member-level; re-publishing still needs `requireRole`) flips the `published` working row back to `draft` (contentVersion +1), links `supersedes` → the latest `publishedReportVersions` snapshot, and clears `approvedBy`/`approvedAt` — the content is unchanged and now editable again. The prior published version remains in the append-only snapshot table, immutable and signed. This is the ONLY escape from published immutability (AC-2) and satisfies "post-publication changes require 'Start new version' creating a new draft superseding record": the working row becomes a new draft that supersedes the archived published record. Re-publishing runs the full `approveAndPublish` gate again (a new snapshot, a new signature).

### §D6 — the client resolution count reuses the server blocker's helper + the chip resolver

`reportCitationResolution` (client) is the display + Approve-disable layer over the SAME `uncitedSentences` the server enforces (D2), plus the chip `resolveDiagnostic` check for dangling-marker honesty. "N claims · N citations resolve" (UX-DR13) = totalClaims (figure-bearing sentences) vs resolvedClaims (claims with a resolving citation). The bar disables Approve when any claim fails and links each failing sentence to its section (UX-DR13 "the failing sentence linked"; reuse the chip hash-scroll). Because the client uses the server's helper, the button state matches the server gate exactly — no client/server divergence, no client-trusted gate (the server re-checks). Pure metadata aggregation (AD-1 clean).

The overrides count comes from the existing `getRecommendationOverrides` subscription (6.3), so no new query.

### §D9 — server-ack-only, and the diff-since-draft baseline

No optimistic UI (AD-3/EXPERIENCE.md "Banned … a logged approval must never flicker"): the bar re-renders from the `getReserveReport` subscription on server ack. "Diff since draft" (UX-DR13) needs a baseline that does not exist today (6.1 overwrites the machine draft in place on edit) — so 6.4 captures it: `editReserveReport` snapshots the machine-drafted original into `draftBaseline` on the FIRST human edit (`machineDrafted` `true → false`), a one-conditional addition. The diff link compares the current `report` against `draftBaseline` per section; a manual report or a never-edited machine draft has no baseline → the link is disabled with "No analyst edits to compare". The diff view is a simple per-section string comparison (display-only, AD-1); a rich word-level diff is a deferrable nicety.

### Versioning vs. superseding (the two `contentVersion` roles)

`contentVersion` is the IN-PLACE edit counter (6.1: 1 at create, +1 per edit) — the version the approver signs. It is DISTINCT from the published-snapshot history: each `publishedReportVersions` row records the `contentVersion` that was signed at that approval. "Start new version" increments `contentVersion` again (the re-opened draft) and links `supersedes` to the last snapshot. So a Run's report history is: the current working row (`reserveReports`, one per Run) + the append-only `publishedReportVersions` chain (0..n signed snapshots). Do NOT conflate them or add a second `version` field to the working row.
