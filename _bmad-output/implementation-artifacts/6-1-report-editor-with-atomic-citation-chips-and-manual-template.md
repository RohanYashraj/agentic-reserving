# Story 6.1: Report Editor with Atomic Citation Chips and Manual Template

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an Analyst,
I want to edit the draft report with citations as atomic tokens — or start from a manual template when Interpretation is down,
so that human judgment shapes the report without breaking its evidence trail. (FR-13 edit, FR-12, UX-DR12)

## Acceptance Criteria

1. **Given** a draft Reserve Report, **When** the Analyst edits in the section-structured editor (executive summary, method selection rationale, movement commentary, limitations), **Then** citation chips are atomic — editable around, never inside — and deleting a chip flags its sentence "claim now uncited", tracked as a blocker for approval (UX-DR12), **And** edits save via a Convex mutation, are audit-logged with actor and content version, and human edits are **not** re-run through the Provenance Gate (AD-5 scope boundary).
2. **Given** Engine-Only Mode (or no Interpretation), **When** the Analyst opens the Report tab, **Then** they can create a Reserve Report shell from the manual template with the same four sections for hand drafting (FR-12), **And** convex-test covers edit permissions (Analyst can edit drafts, no one edits a non-draft/published report) and audit entries.

## Scope decisions baked into this story (read before coding)

These are deliberate calls the dev agent must honor, not open questions. Rationale in Dev Notes §D1–D9. The two genuinely-open questions for Rohan are listed at the very end (§Questions for Rohan) — they do **not** block starting.

- **D1 — The editor is a structured token-model, NOT `contentEditable` and NOT a raw-marker textarea.** Each section is edited as an ordered list of nodes — `{ kind: "text", value } | { kind: "cite", dxId }` — parsed from the stored `section.text` by splitting on the inline `[[cite:<dxId>]]` markers the Provenance Gate rendered (§What already exists — the marker grammar). Text nodes are editable; chip nodes are **atomic widgets** rendered via the existing `CitationChip`, so "editable around, never inside" is true *by construction* (a chip is a React widget, not editable text). On save the node model serializes **back** to `{ text (with `[[cite:...]]` markers), citations[] (re-derived from the markers, document order) }`. This is the zero-dependency, jsdom-testable, React-safe choice that matches the house style (no editor library is installed; the repo hand-rolls — CitationChip preview, banner toast). `contentEditable` (higher prose fidelity, "chips flow inside one field") is the rejected higher-risk alternative — flagged in §Questions for Rohan and deferred-work. (See §D1.)
- **D2 — Human edits write the SAME `reserveReportValidator` shape; `citations[]` is re-derived from the markers server-side, never trusted from the client.** The `editReserveReport` mutation takes the four edited section **texts** (each containing `[[cite:...]]` markers), re-derives each section's `citations[]` by scanning its markers (a pure string scan — no arithmetic, no gate, AD-1 clean), and stores the typed document. Deleting a chip therefore *automatically* drops its citation from `citations[]` and keeps `text`↔`citations` consistent forever. The mutation validates marker **syntax** structurally (well-formed `[[cite:dx:...]]`) so the stored text stays tokenizable — this is integrity, **NOT** the Provenance Gate (AD-5: human edits are never re-gated). (See §D2.)
- **D3 — "Claim now uncited" is a DERIVED display flag over the stored text, not a stored boolean.** A "claim" for the editor flag is a **sentence** containing an engine-figure numeric token but **no** `[[cite:...]]` marker. It is computed from `section.text` (display-metadata aggregation, AD-1 clean — same posture as 5.5's "cited by N"). 6.1 ships the derivation helper `uncitedSentences(text)`; the editor renders the inline flag, and Story 6.4's approval bar consumes the same helper for its "N claims · N citations resolve" count + "failing sentence linked" blocker. No new stored field, no server round-trip. Sentence granularity is finer than the gate's paragraph-block claim model (deferred-work §5.2) — documented divergence. (See §D3.)
- **D4 — 6.1 adds the content-versioning + human-ownership columns the 5.4 table deliberately left out.** `reserveReports` gains `contentVersion: number` (starts `1` at machine draft / manual create; `+1` on every human edit), `updatedBy: string`, `updatedAt: string`. The `machineDrafted` boolean flips to `false` on the first human edit (the current version is now human-owned, AD-5). The audit entry for an edit carries `{ runId, contentVersion }` — the "content version" the AC and AD-5 name (the approver signs a specific `contentVersion` in 6.4). (See §D4.)
- **D5 — 6.1 extends `status` to the full lifecycle enum so the immutability guard is real and testable now, but only WRITES `draft`.** `status` becomes `v.union(v.literal("draft"), v.literal("awaiting_review"), v.literal("published"))`. 6.1 writes only `draft` (manual create → draft; edit keeps draft). The transitions to `awaiting_review` (6.2) and `published` (6.4) are those stories'. Extending now lets `editReserveReport` guard `status === "draft"` (reject otherwise → `REPORT_NOT_EDITABLE`) and lets AC-2's "no one edits a non-draft/published report" be a **real** convex-test (seed a `published` row, assert the edit is rejected). (See §D5.)
- **D6 — The machine-draft generation trigger lands HERE, on the Report tab (resolving 5.5's deferred D1).** 5.5 deliberately made "Generate interpretation" trigger `generateRecommendations` ONLY and deferred "should it chain the report draft?" to Epic 6. 6.1 resolves it: the **Report tab** carries a "Generate report draft" trigger that calls the existing `generateReserveReport` action (5.4) when the Run has accepted recommendations and the Workspace is NOT Engine-Only. "Generate interpretation" (Interpretation tab) stays recommendations-only. Without this, AC-1's "Given a draft Reserve Report" has no product path to a draft. Reuse `InterpretationTab`'s trigger-state pattern; do NOT chain the two actions. (See §D6.)
- **D7 — The manual template is offered when NO report exists (Engine-Only, or Interpretation simply not run).** `createManualReport` builds a `draft` with four **empty** sections (`text: ""`, `citations: []`), `machineDrafted: false`, `contentVersion: 1`. Guard: only creatable when the Run has no `reserveReports` row (`REPORT_ALREADY_EXISTS` otherwise). It needs no `DiagnosticsBundle` (empty sections have no chips). The Report tab offers it whenever no report exists; in Engine-Only Mode it is the ONLY path (the D6 generate trigger is disabled with a tooltip, mirroring 5.6's InterpretationTab). The Engine-Only banner already points here ("Report drafting from a manual template arrives with the report editor", `EngineOnlyBanner.tsx`). (See §D7.)
- **D8 — Chips in the editor stay navigable AND gain an edit-mode delete affordance; the manual/Engine-Only path tolerates a missing bundle.** In the editor, each `CitationChip` keeps its 5.5 behavior (violet pill, click/Enter → Diagnostics deep-link, hover/Space → preview) and, in edit mode, gains a small adjacent "remove" control (an `×` button, NOT inside the chip) that deletes the chip node. `CitationChip` already degrades when `resolveDiagnostic` returns `null` (stays navigable, falls back to the raw-id aria-label) — the manual/Engine-Only path (no bundle, or `diagnosticsBundle == null`) reuses that fallback; a manual report has no chips anyway. Provenance-violet stays exclusive to chips (DESIGN.md:89). (See §D8.)
- **D9 — The count-of-citing-claims tally 5.5 left open now unions the report's section citations.** 5.5's `countCitingClaims` (context rail "cited by N report claims") counts recommendation-reason citations only, with a documented note that "Epic 6 unions the Reserve Report's section citations". 6.1 threads the report's per-section `citations[]` into that tally so a Diagnostic cited by a report section increments its "cited by N" backlink. Keep it citation-metadata aggregation (AD-1 clean); do NOT add a Convex query — the report is already subscribed on the page. Wiring the "→ view in draft" backlink navigation (mockup) is optional this story; if omitted, flag it. (See §D9.)

### Architecture compliance (non-negotiable)

- **AD-1 / The Constitution — the LLM never calculates, and neither does the UI, and neither does a human typing.** Every figure in a report is an engine value the gate rendered from a `{{rs:...}}` placeholder (5.4); the React editor does **display formatting only**. The marker tokenizer, `citationsFromText`, and `uncitedSentences` are **string/metadata** operations (no sums/deltas/ratios) — AD-1 clean, the same posture as 5.5's "cited by N". A human edit re-derives `citations[]` by scanning markers, never by computing anything. 6.1 does NOT let a human insert a NEW engine figure by typing a number — figures already live in the machine-drafted prose as rendered values with their chips; manual-template hand drafting is qualitative prose (a hand-typed number is the author's own words, not an engine-sourced reserve figure, and carries no chip — it will simply flag as an uncited claim if it looks like a figure, which is the honest signal). (See §Questions for Rohan on manual-mode figures.)
- **AD-5 — the Provenance Gate governs machine-drafted content ONLY; human edits are human-owned and NOT re-gated.** This is the load-bearing rule of 6.1 (spine AD-5 scope boundary, verbatim: "the gate governs machine-drafted content only; subsequent human edits are human-owned and audit-logged, and the approver signs the exact content version (FR-13) — the gate is not re-run on human edits"). `editReserveReport` does NOT call `run_provenance_gate` / any engine endpoint — it re-derives citations and stores. The only edit-time validation is **marker syntax integrity** (keep the text tokenizable), which is NOT the gate. The evidence trail is preserved because chips stay atomic `[[cite:<dxId>]]` markers carrying the full resolvable Diagnostic ID (AD-10); a human can delete a chip (orphaning a claim → the honest "uncited" flag) but cannot forge one (a `×` removes; adding requires a real marker, and a bogus id resolves to nothing → the chip degrades + the sentence still can't cite it meaningfully).
- **AD-6 / FR-15 — every report edit is audit-logged.** `report.edited` (actor + `contentVersion`) and `report.manualCreated` (actor) via the sole writer `appendAuditEntryInTransaction`; the chain stays append-only + hash-chained. "The approver signs the exact content version" (AD-5/FR-13) is why `contentVersion` is a first-class column now (6.4 records the signed version; 6.1 mints/increments it). Lean payloads — NO section text/figures (the content lives on the row).
- **AD-4 — no anonymous access; role split.** `editReserveReport`/`createManualReport` call `requireMember` first (editing/creating a draft is a member action). Approve/publish/override are `requireRole(senior_actuary)` in 6.3/6.4 — NOT here. Both new public functions are enumerated in `authGuard.test.ts` (Task 4). Server-side enforcement, never UI-only (the `status:"draft"` editability guard is in the mutation, not just the tab).
- **AD-9 — Engine-Only Mode; the manual shell stays functional.** `createManualReport` has NO interpretability precondition (no resultSet/bundle/recommendations required) — hand drafting works during a model outage (AD-9: "manual report drafting from the template shell remain fully functional"). Engine-Only Mode is read from the server-derived `getInterpretationMode` (never a client guess); the generate-draft trigger is disabled-with-tooltip while `engineOnly`, the manual path is primary. NFR-2 invariant (upload → Run → Diagnostics fully functional) is untouched — 6.1 adds only the Report tab.
- **AD-10 — Diagnostic ID is `dx:{runId}:{kind}:{key}`, resolvable, never string-split.** The `[[cite:...]]` marker carries the FULL id; the tokenizer treats it as opaque (matches `dx:[^\]]+`, never parses the internals); `CitationChip` resolves via the shared index. The report document stays the drift-checked engine contract — human edits do not alter its shape (Task 1.3).

No optimistic UI on save (EXPERIENCE.md:101-102) — the state flips on server ack.

### §D1 — the token-model editor (why not contentEditable, why not a raw textarea)

Three ways to render "flowing prose with inline atomic chips, editable around not inside":
1. **`contentEditable`** with `contentEditable={false}` chip spans — the highest prose fidelity ("chips flow in one field, travel with the sentence"), but notoriously hard in React (selection/IME/reconciliation, Backspace-into-widget edge cases) and the repo has ZERO `contentEditable` and a strong hand-rolled/zero-dep bias.
2. **Raw-marker `<textarea>`** — edit `text` including literal `[[cite:...]]` as plain text + a live preview. Simplest, but a chip is NOT atomic in the editing surface (a user can edit *inside* `[[cite:dx:...]]`), directly violating UX-DR12 "editable around, never inside."
3. **Structured token-model (CHOSEN)** — sections are an ordered `SectionNode[]`; text nodes are editable fields, chip nodes are `CitationChip` widgets. Atomicity is **structural** (a chip is a widget, not text — there is no caret "inside" it), zero-dep, jsdom-testable, React-native, and serializes cleanly back to the stored `text`+`citations` contract. The cost is a segmented editing feel (editing happens in the text runs between chips) rather than one flowing field — an acceptable, honest trade for correctness and low risk. This is the recommended default; contentEditable is flagged for Rohan (§Questions) and deferred-work as the fidelity upgrade if the segmented feel proves awkward with real drafts.

### §D2 — server re-derives citations; the client is never trusted for the pin

The machine-readable `citations[]` is the "pin" downstream code (the "cited by N" tally, 6.4's resolution count) relies on. If the client sent both `text` and `citations`, they could drift (a malicious/buggy client could claim a citation the text doesn't contain). So `editReserveReport` takes ONLY the four texts and **re-derives** `citations[]` from the `[[cite:...]]` markers server-side (`citationsFromText`). Deleting a chip in the editor removes the marker from the serialized text → the server-derived `citations[]` omits it automatically. `text`↔`citations` are consistent by construction, forever. The mutation validates marker *syntax* (integrity, keep it tokenizable) — this is explicitly NOT the Provenance Gate (AD-5: no re-gate on human edits); it does not check numeric provenance or claim-citation coupling (those were the gate's job on the machine draft; a human orphaning a claim is the *intended* "uncited" flag, not a rejection).

### §D3 — "claim now uncited" is a derived display flag (sentence granularity)

The AC's "deleting a chip flags the sentence 'claim now uncited', tracked as a blocker for approval" is a **display derivation**, not stored state: `uncitedSentences(text)` returns sentences that state a figure but carry no `[[cite:...]]` marker. The editor renders the inline flag; 6.4's approval bar consumes the SAME helper for "N claims · N citations resolve" + the disabled-Approve "failing sentence linked" (UX-DR13). One source of truth, no stored boolean, no server round-trip (citation-metadata over the already-subscribed report — AD-1 clean). Granularity is the **sentence** (finer than the gate's paragraph-block claim model, deferred-work §5.2) because UX-DR12 says "flags the *sentence*"; documented so a reviewer doesn't "reconcile" it to block granularity. Resolution paths (EXPERIENCE.md:160): re-cite the claim, or delete the orphaned sentence — both clear the flag naturally (the sentence no longer has a figure-without-chip).

### §D4 — content versioning vs. superseding (two different Epic-6 concepts)

`contentVersion` (this story) is the **in-place edit counter** on the single draft row — it increments each human edit and is the "content version" the approver signs (AD-5/FR-13). It is NOT Epic 6.4's "start new version" **superseding record** (FR-13, spine ERD `REPORT ||--o{ REPORT : "new version supersedes"`), which creates a NEW `reserveReports` row after a *published* report is reopened. 6.1 owns `contentVersion`; 6.4 owns supersession (and may add `supersedes`/`version` columns then). Keeping them distinct avoids conflating "I edited the draft twice" (contentVersion 1→3) with "the published report was reopened as a new version" (a new row). Documented so 6.4 extends cleanly.

### §D6 — the report-draft generation trigger lives on the Report tab

5.5 deferred (D1) whether "Generate interpretation" should chain `generateReserveReport`. 6.1 resolves it: NO chaining. "Generate interpretation" (Interpretation tab) triggers recommendations only; the **Report tab** carries a separate "Generate report draft" trigger calling `generateReserveReport` (5.4) — shown when the Run has accepted recommendations and the Workspace is not Engine-Only. Rationale: (a) it keeps each tab's trigger owning its own deliverable (recommendations vs. report), (b) it avoids the partial-success failure mode of one combined action, (c) it gives AC-1 a real product path to "a draft Reserve Report", and (d) the manual template is the always-available alternative. The generate trigger reuses `InterpretationTab`'s pending/rejected state pattern (skeleton + quiet "Drafting…"; gate-exhaustion shows the quiet failure copy — 5.4 returns `{status:"rejected"}` cleanly).

### §D7 — the manual template shell

`createManualReport` builds a `draft` with four empty sections, `machineDrafted:false`, `contentVersion:1`, and NO interpretability precondition — it must work in Engine-Only Mode (AD-9). It is offered whenever no `reserveReports` row exists for the Run (guarded `REPORT_ALREADY_EXISTS` on a second call). In Engine-Only Mode it is the only path (the generate trigger is disabled-with-tooltip via the 5.6 `getInterpretationMode` + `components/ui/tooltip.tsx` idiom); otherwise it is a secondary "Start from a blank template" beside the generate trigger. The four section names are canonical/fixed (`executiveSummary`, `methodSelectionRationale`, `movementCommentary`, `limitations`). A manual report has no chips (empty `citations[]`), so the atomic-chip/uncited machinery is inert until the Analyst hand-types content.
