# Story 5.5: Interpretation Tab with Recommendation Table and Citation Chips

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an Analyst,
I want to trigger Interpretation and review the recommendation table with working citation chips,
so that the reasoning is transparent and navigable, not a black box. (FR-10 UI, UX-DR2, UX-DR11, UX-DR16)

## Acceptance Criteria

1. **Given** a complete Run, **When** the Analyst clicks "Generate interpretation", **Then** the Interpretation tab shows a skeleton recommendation table with "Reading diagnostics…" (no token streaming; output appears complete after the gate passes), and gate retries show the quiet status "Draft failed provenance check — redrafting (attempt N of M)" (UX-DR16), **And** the panel carries the header "Drafted by the interpretation layer · every claim cites a diagnostic".
2. **Given** accepted output, **When** the table renders, **Then** each Origin Period row shows the recommended Method and reasons trailing CitationChip components — provenance-subtle pill, Diagnostic ID in numeric type; hover fills violet with a tooltip preview of the cited value; click navigates to `/runs/{id}/diagnostics#<dxId>`; chips are tab-stops where Enter navigates and Space previews, announced as links with context (UX-DR2).
3. **And** the Diagnostics context rail "cited by N report claims" backlinks now populate (completing Story 4.6's contract).
4. **And** interpretation status updates arrive via subscription, and triggering/completion/failure events are audit-logged.

> **This is a FRONTEND + thin-Convex-audit story.** The entire interpretation backend already exists and ships accepted, drift-checked, and tested: the `generateRecommendations` action, the `getRecommendations` query, the `hasRecommendations` boolean on `getRun`, the `recommendations` inline field on `runs`, and the recommendations contract (`recommendationsValidator` → `MethodRecommendation` / `RecommendationReason`) with `reasons[].citations` as resolved `dx:` IDs. **Do NOT re-implement or modify the recommendation-generation backend, the Provenance Gate, or the recommendations schema.** 5.5 wires the UI onto those forward-declared contracts and adds ONE audit event (`run.interpretationTriggered`). See Dev Notes §What already exists.

## Scope decisions baked into this story (read before coding)

These are deliberate calls the dev agent must honor, not open questions. Rationale in Dev Notes.

- **D1 — "Generate interpretation" triggers `generateRecommendations` ONLY.** The recommendation table is 5.5's visible deliverable. Chaining the Reserve Report draft (`generateReserveReport`, 5.4) is **NOT** in 5.5 — the Report tab renders in Epic 6. (See §D1.)
- **D2 — No live "attempt N of M" counter.** The interpretation loop runs entirely inside ONE synchronous engine HTTP call (5.3/5.4 architecture; the async `202 + callback` transport is deferred, spine:204/209). The engine's per-attempt redraft progress is therefore **not observable** by the client mid-call. 5.5 renders the achievable UX-DR16 states: skeleton + "Reading diagnostics…" during the pending action (client `useAction` pending, `aria-live="polite"`), and on a **rejected** outcome the quiet failure "Draft failed provenance check — the interpretation could not be drafted (all attempts failed the provenance check)." The parametric live "attempt N of M" ticker is deferred with the async transport (flagged in deferred-work). (See §D2.)
- **D3 — CitationChip renders the canonical `dx:{runId}:{kind}:{key}` ID in numeric mono**, exactly like the existing `DiagnosticId.tsx` panel chip — NOT a fabricated "D-LDF-07" friendly label (the UX mockups' `D-LDF-07` is display shorthand with no numbering scheme in the data; 4.6 already deferred the friendly-label decision). The accessible name supplies the human context. (See §D3.)
- **D4 — "cited by N report claims" counts recommendation-reason citations, client-side.** N = number of `RecommendationReason`s (across all Origin Periods) whose `citations[]` include the selected Diagnostic ID, computed in React from the `getRecommendations` subscription (citation-metadata aggregation, NOT reserve arithmetic — AD-1 clean). No new Convex query. The "view in draft" link and report-section citations are Epic 6. (See §D4.)
- **D5 — The per-row "accepted / overridden" status column and the Senior-Actuary Override action/dialog are Epic 6.3** (`requireRole(senior_actuary)`, UX-DR11). 5.5 renders recommendation + reasons + chips only. (See §D5.)
- **D6 — Navigation reuses Story 4.6's hash mechanism verbatim.** A chip click/Enter sets `window.location.hash = <dxId>`; `RunDetail`'s existing `hashchange` effect flips to the Diagnostics tab and `DiagnosticsPanels` selects + scrolls + highlights. The full `/runs/{id}/diagnostics` **path segment** stays deferred (4.6 deferred-work) — the `#<dxId>` anchor contract is what the chip fulfils. (See §D6.)

### Architecture compliance (non-negotiable)

- **AD-1 / The Constitution — the LLM never calculates, and neither does the UI.** Every number the tab shows is a value the engine produced and the Provenance Gate already rendered into `reason.text` (5.3) — the React layer does **display formatting only** (`ARCHITECTURE-SPINE.md:57`). No sums/deltas/ratios anywhere in 5.5. The "cited by N" count is aggregation of **citation metadata** (how many reasons name an id), not reserve arithmetic — permitted. The chip preview reads stored Diagnostic values verbatim.
- **AD-3 / AD-7 / FR-20 — Convex is the reactive system of record.** The tab subscribes to `getRun`/`getRecommendations`/`getDiagnosticsBundle` via `useQuery` (live, no polling). Interpretation status = the durable subscription (`hasRecommendations` flips → table populates). The transient "Reading diagnostics…" is client `useAction` pending — acceptable because the durable outcome is the subscription. **No polling in application code** (FR-20; `ARCHITECTURE-SPINE.md:198`, `prd.md:217`).
- **AD-4 — no anonymous Convex access.** No new public function is added (the trigger uses the existing `generateRecommendations`, already `requireMember`-guarded + enumerated). `recordInterpretationTriggered` is `internalMutation`. The `page.tsx` gate on `orgId` before subscribing is the existing pattern.
- **AD-5 — the UI renders only what the gate passed.** Every reason carries ≥1 resolvable `dx:` citation by the 5.3 contract; "a claim without a chip cannot exist on screen" (EXPERIENCE.md:55). 5.5 does not re-validate — it trusts the gate + the typed `getRecommendations` surface.
- **AD-6 / FR-15 / NFR-5 — triggering, completion, failure audit-logged.** Completion (`run.recommended`) + failure (`run.interpretationRejected`) exist; 5.5 adds **triggering** (`run.interpretationTriggered`) via the sole writer `appendAuditEntryInTransaction`. Chain stays append-only + hash-chained (`AD-6`).
- **AD-9 — fail-closed is Story 5.6, not 5.5.** 5.5 surfaces `engine.model_unavailable` / transient failures as a readable inline message; the full-bleed Engine-Only banner, the disabled-with-tooltip action, and the derived-server-side mode are **Story 5.6**. Do NOT build the banner or a client-side mode guess here (AD-9 — "never a client-side guess").
- **AD-10 — Diagnostic ID is `dx:{runId}:{kind}:{key}`, resolvable, never string-split.** The chip renders and navigates on the canonical id; resolution is via the shared index (`getElementById` for the DOM target, the bundle index for the preview) — never a CSS `querySelector('#…')` (colons) and never by splitting the id to fabricate a label (D3).

### §D1 — "Generate interpretation" triggers recommendations only

PRD glossary defines Interpretation as "per-Origin-Period Method recommendation with reasons, **and** the draft Reserve Report" (`prd.md:57`). But 5.5's ACs and its visible deliverable are the **recommendation table + chips** (`epics.md:645-649`); the Report renders in Epic 6, and 5.4 already ships `generateReserveReport` as a separate action. Chaining report drafting into this trigger would (a) render nothing new in 5.5 (the report tab is Epic 6), (b) introduce a partial-success failure mode (recommendations pass, report fails), and (c) roughly double interpretation latency for no 5.5-visible benefit. So 5.5 triggers `generateRecommendations` only; the report-draft chain (or a combined "Generate interpretation" that produces both) is an Epic-6 decision — flagged in deferred-work. Consequence for D4: "cited by N" counts recommendation-reason citations (the interpretation claims that exist after 5.5's trigger).

### §D2 — why there is no live "attempt N of M"

The bounded redraft loop runs **inside `engine_service`, within one synchronous HTTP call** from the Convex action (5.3/5.4 design; `recommendations_flow.generate_recommendations`). The Convex action `await`s that single `callEngine("/recommendations")` and returns only after the engine has exhausted or accepted — the per-attempt progress never leaves the engine process mid-call. The architecture explicitly defers the transport that would expose it: `ARCHITECTURE-SPINE.md:93` shapes the contract for "an async `202 + HMAC-signed callback` upgrade [that] is additive", and `:204`/`:209` defer streaming/durable-transcript-under-crash. Therefore a faithful live "attempt 2 of N" is **not implementable in 5.5** without that deferred transport (and even a durable Convex `@convex-dev/workflow` wrapper couldn't see the engine's internal attempts without the callback). Fabricating a client-side ticker (à la `UploadWizard`'s choreographed stages, deferred-work §3 line 102) would show progress the client cannot verify — rejected here for honesty. 5.5 renders: skeleton + "Reading diagnostics…" while pending, and the quiet gate-failure copy on a `rejected` outcome. The parametric live ticker is deferred with the async transport.

### §D3 — canonical id, not a friendly label

The data carries `dx:{runId}:{kind}:{key}` (AD-10); the UX mockups show a display shorthand `D-LDF-07` (`EXPERIENCE.md:70`, `DESIGN.md:126`) with **no corresponding numbering scheme in the engine output** (the `07` is illustrative). The existing `DiagnosticId.tsx` and the context rail both render the **full canonical id** in provenance-violet `numeric` mono — 5.5's CitationChip matches that established convention for consistency + guaranteed resolvability. The human context UX-DR2 wants ("announced as … loss development factor stability") lives in the `aria-label` + the tooltip preview, not in a fabricated visible label. A friendly-label transform (`kind` + coordinate → a readable string) is deferred (4.6 already deferred the friendly-label decision; deferred-work).

### §D4 — "cited by N" is client-side citation-metadata aggregation

The count is `recommendations.recommendations.flatMap(r => r.reasons).filter(reason => reason.citations.includes(selectedId)).length`. This reads only the citation id lists (opaque strings), touches no figure, and is display-side — AD-1 clean, no new Convex query, no server round-trip (the rail already has `recommendations` via props from the same page that subscribes). Precompute a `Map<dxId, count>` once per recommendations change if you prefer O(1) per selection. "report claims" is the UX label for interpretation claims; Epic 6 will union the Reserve Report's section citations into the same count (documented at the call site).

### §D5 — override column is Epic 6.3

UX-DR11 (`epics.md:90`) specifies the recommendation table also carries a per-row **status (accepted/overridden)** and a Senior-Actuary **Override action → reason dialog**, with recommendation + override shown side-by-side "history never erased". That is a `requireRole(senior_actuary)` write path — **Epic 6 Story 6.3** (`epics.md:713-724`, same UX-DR11 tag). In 5.5 there are no overrides yet, so a status column would be uniformly "accepted" noise. 5.5 renders recommendation + reasons + chips; the status column and override UI land with 6.3.

### §D6 — navigation is the 4.6 hash contract

The chip sets `window.location.hash = dxId`. `RunDetail`'s `hashchange` effect (`RunDetail.tsx:102-117`) reads the raw fragment, `setTab("diagnostics")`, and latches `initialSelectedId`; `DiagnosticsPanels` (`51-73`) resolves it via `getElementById` (colons in the id forbid CSS selectors), selects it into the rail, scrolls, and flashes the highlight. This is exactly "the navigation target citation chips will use product-wide" 4.6 built for. The chip must NOT reach into RunDetail's tab state — the hash is the decoupling seam. The full `/runs/{id}/diagnostics` **route segment** stays hash-only (4.6 deferred-work line 41); the `#<dxId>` anchor contract is what AC-2 requires.
