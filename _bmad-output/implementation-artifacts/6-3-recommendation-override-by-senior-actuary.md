# Story 6.3: Recommendation Override by Senior Actuary

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a Senior Actuary,
I want to override any Method recommendation with a recorded reason,
so that my professional judgment is first-class and traceable. (FR-10 override, UX-DR11)

## Acceptance Criteria

1. **Given** the recommendation table, **When** a Senior Actuary clicks Override on a row, **Then** a dialog requires choosing the overriding Method and entering a reason before confirming, restating that the override will be logged (UX-DR14), **And** the server enforces `requireRole(..., "senior_actuary")` — an Analyst's override attempt is rejected server-side and the control renders visible-but-disabled with tooltip for Analysts (UX-DR18).
2. **Given** a confirmed override, **When** the table re-renders, **Then** recommendation and override display side by side, both attributed, citations intact, history never erased (UX-DR11), **And** the override with reason, actor, and timestamp lands in the Audit Log (FR-10).

## Scope decisions baked into this story (read before coding)

These are deliberate calls the dev agent must honor, not open questions. Rationale in Dev Notes §D1–D9. The genuinely-open questions for Rohan are at the very end (§Questions for Rohan) — they do **not** block starting.

- **D1 — Overrides live in a NEW product-plane child table `recommendationOverrides` (append-only). The engine-contract `recommendationsValidator` on `runs.recommendations` is NEVER mutated (AD-10).** The machine recommendations stay immutable on the run row; an override is a NEW inserted row `{ workspaceId, runId, origin, overridingMethod, reason, overriddenBy, overriddenAt }`. This is the direct answer to §5.3's deferred "if Epic 6 needs recommendations versioned independently of the Run (draft vs overridden), revisit then" (`deferred-work.md §5.3`) — the answer is a separate table, mirroring the `reserveReports` separate-artifact precedent (`schema.ts:204`). "History never erased" (UX-DR11) is therefore **literal on the data plane** (append-only inserts, no patch/delete of an override row) AND in the audit chain (AD-6). Do NOT add an `override`/`overriddenBy` field to `methodRecommendationValidator` — that is the drift-checked engine contract (`engineContract.ts:192-209`, AD-10) and a re-run of `generateRecommendations` overwrites the whole `runs.recommendations` document (it must not carry human state). (See §D1.)
- **D2 — `overrideRecommendation` is the FIRST production caller of `requireRole(ctx, workspaceId, "senior_actuary")` (AD-4 role split).** Every prior public function is `requireMember`; approve/publish/override are the Senior-Actuary-gated paths (`guards.ts:64`, currently zero callers — only anticipatory comments at `runs.ts:1126-1127,1464,1655,1776-1777`). `requireRole` calls `requireMember` internally then throws `ConvexError { code: "FORBIDDEN", message: "This action requires the senior_actuary role." }` when `member.role !== "senior_actuary"` (`guards.ts:70-75`). This IS the server-side rejection of an Analyst override (AC-1) — never UI-hiding alone (AD-4). (See §D2.)
- **D3 — The reason is required, human-owned, and NOT re-run through the Provenance Gate (AD-5); it IS recorded in the audit payload (FR-10).** An override is a human judgment, not machine-drafted content — `overrideRecommendation` calls NO engine endpoint / gate (AD-5: "the gate governs machine-drafted content only; human edits are human-owned and audit-logged"). The mutation guards a non-empty trimmed `reason` (`REASON_REQUIRED` else) — this is input validation, NOT the gate. Unlike the lean `report.*` audits (which deliberately exclude section text), the `recommendation.overridden` audit payload **carries the reason verbatim** because FR-10's testable consequence is explicit: "the override and its recorded reason land in the Audit Log." The reason is short human prose (the author's own words), never an engine figure — AD-1 is not in tension (no arithmetic, no reserve figure originates here). (See §D3.)
- **D4 — The dialog's Method picker offers the methods ≠ the current recommendation; re-override is allowed (append, latest-per-origin wins).** The overriding Method is one of the three `methodValidator` literals (`chain_ladder | bornhuetter_ferguson | mack`, `engineContract.ts:25-29`) OTHER than the recommended one (overriding to the already-recommended method is a no-op the picker excludes; the server also rejects `overridingMethod === recommendedMethod` → `OVERRIDE_MATCHES_RECOMMENDATION`). A Senior Actuary may override an already-overridden row (change their mind): each confirm INSERTS a new row; the display shows the LATEST override (max `overriddenAt`) for that origin beside the ORIGINAL machine recommendation (never the prior override); every prior override remains in the table's history query + the audit log (D1 "history never erased"). (See §D4.)
- **D5 — The per-row status tag (`accepted` / `overridden`) is a lightweight INLINE label, NOT the `StatusBadge` component.** `StatusBadge` owns the FIXED Run/Report lifecycle vocabulary (`draft|queued|running|complete|failed|awaiting review|published|engine-only`, `StatusBadge.tsx:28-36`, UX-DR3) — do NOT add `accepted`/`overridden` to it (they are recommendation-row states, a different axis). Render a small inline pill in the table using the same token families (`overridden` → `bg-caution-subtle text-caution`, the "needs your judgment" amber; `accepted` → neutral `bg-muted text-muted-foreground`) — NO hard-coded hex (DESIGN.md). (See §D5.)
- **D6 — The override dialog REUSES the 6.2 `Dialog` primitive (`components/ui/dialog.tsx`) and mirrors `ReportApprovalBar`'s `DraftBar` dialog pattern exactly (UX-DR14).** `onOpenAutoFocus` → initial focus on **Cancel** (`ReportApprovalBar.tsx:156-159`); a `useState` pending flag disables both footer buttons and swaps the confirm label ("Overriding…"); NO optimistic UI — the confirm awaits the mutation and the `getRecommendationOverrides` subscription re-renders the row on server ack (D9); `onOpenChange` clears the inline error on dismiss and Cancel/Esc fire the action NEVER (`:138-143`). Zero new dependency — the primitive already exists. (See §D6.)
- **D7 — Client `canOverride` is derived from `useAuth().orgRole` + `normalizeRole`; UX-DR18 renders the Analyst's control visible-but-disabled with a Tooltip. Server authority stays `requireRole`.** 6.3 introduces the FIRST current-user-role client signal (today only `seniorActuaries`, a LIST of members, is built client-side — `page.tsx:101-116` — no "is the CURRENT user a Senior Actuary" flag exists). Add `const { orgId, orgRole } = useAuth();` → `canOverride = normalizeRole(orgRole) === "senior_actuary"` and thread it page → `RunDetail` → `InterpretationTab` → `RecommendationTable`. For a Senior Actuary the Override button is live; for an Analyst it is a disabled button inside a focusable `<span tabIndex={0}>` wrapped in a Tooltip reading "Senior Actuary role required" (the exact 5.6/6.1 disabled-action idiom, UX-DR18). Client gating is display only — the server `requireRole` is the enforcement (AD-4; the convex-test proves an Analyst call is rejected regardless of UI). (See §D7.)
- **D8 — Side-by-side display: the ORIGINAL recommendation (method + reasons + atomic chips) stays intact; the override renders beside it as an attributed card (method, quoted reason, `overriddenBy · Senior Actuary · {timestamp}`).** Mirror mockup `report-review.html:64-68` (a `border-left:3px solid var(--primary)` teal-accent card with the `CL → BF` transition, the quoted reason, and the `.who` attribution line). The recommendation's `CitationChip`s are passed the SAME `dxId`/`diagnosticsBundle` and never re-parsed — they stay atomic/navigable across the override re-render (AC-2 "citations intact"). The override card carries NO chips (a human reason cites no Diagnostic — it is professional judgment). (See §D8.)
- **D9 — New subscription `getRecommendationOverrides`, gated on `run.hasRecommendations`; reactive, no polling, survives reload (AD-3/FR-20).** A `requireMember` Workspace-scoped query returning the override rows for a Run; the page subscribes with `useQuery` gated exactly like `getRecommendations` (`page.tsx:56-59`). On override confirm the mutation inserts a row → the subscription re-emits → the table re-renders with the override, no optimistic UI, reload-durable. No `getRun.hasRecommendationOverrides` boolean is needed (overrides only exist when recommendations do — gate on the existing `hasRecommendations`). (See §D9.)

### Architecture compliance (non-negotiable)

- **AD-4 — no anonymous access; the role split is now REAL.** `overrideRecommendation` is `requireRole(ctx, workspaceId, "senior_actuary")` FIRST — the first production use of the role gate (approve/publish/override are the SA-gated paths, spine AD-4 verbatim). An Analyst's override is rejected SERVER-SIDE (`FORBIDDEN`), never by UI-hiding — the visible-but-disabled control (UX-DR18/D7) is a courtesy, the convex-test proves the server rejection independent of any UI. `getRecommendationOverrides` is `requireMember` (viewing overrides is member-level; only the write is privileged). Both new public functions are enumerated in `authGuard.test.ts` (Task 3).
- **AD-6 / FR-10/FR-15 — every override is audit-logged, history never erased.** `recommendation.overridden` (actor + `origin` + `recommendedMethod` + `overridingMethod` + **reason**) via the sole writer `appendAuditEntryInTransaction`; the chain stays append-only + hash-chained. The reason IS in the payload (D3 — FR-10's explicit testable consequence). "History never erased" (UX-DR11) is doubly guaranteed: the `recommendationOverrides` table is append-only (no patch/delete, D1) AND the audit chain is immutable (AD-6). The spine's AD-6 event list names "override" explicitly.
- **AD-5 — the gate governs machine-drafted content only; an override is human-owned, NEVER re-gated.** `overrideRecommendation` calls NO engine endpoint / Provenance Gate — it inserts a human judgment and audit-logs it (AD-5: "human edits are human-owned and audit-logged … the gate is not re-run"). The only validation is input integrity (non-empty reason, method ≠ recommendation, origin exists) — NOT the gate. The machine recommendation's citations/reasons are untouched (they stay the gated machine content beside the human override — the side-by-side of AC-2).
- **AD-10 — the engine contract is untouched.** Overrides are a product-plane table (`recommendationOverrides`), NOT part of the drift-checked `recommendations` document (`recommendationsValidator`, Task 1.2). The schema-drift CI check and `tests/engine-contract.test.ts` stay green. A machine re-run overwrites `runs.recommendations` and never sees the overrides table (D1) — human state and machine state are cleanly separated.
- **AD-1 / The Constitution — no arithmetic; display-only.** The override mutation, the overrides query, and the table render do NO arithmetic on reserve figures — they record a chosen Method (a categorical label) + a reason string and render them. The override card shows a `method → method` categorical transition and human prose — no figure originates or changes here. The recommendation's figures live in its reasons' rendered `text` (gated machine content, unchanged). The reason is human words, never an engine value (D3).
- **AD-3 / FR-20 — reactive system of record; no polling; no optimistic UI for audit-generating actions.** The table derives overrides purely from the `getRecommendationOverrides` subscription; the dialog flips nothing locally — it awaits the mutation and lets the subscription re-render the overridden row on server ack (AC-2, UX-DR14, D6/D9). Reload survives (durable table row). This is the load-bearing UX rule of the override (an "explicit, logged" judgment).

### §D1 — a separate append-only table, not a field on the engine document

The machine recommendations are the drift-checked engine contract (`recommendationsValidator`, AD-10) and are overwritten wholesale by a re-run of `generateRecommendations` — they must carry NO human state or a re-run would clobber it (and the schema-drift check would fail if the contract grew a human field). So an override is a row in a NEW product-plane table `recommendationOverrides`, keyed by `(runId, origin)`, inserted append-only. "History never erased" (UX-DR11) is then literal on the data plane: an override is never patched or deleted, a re-override inserts a second row, and the LATEST per origin is the current one for display (D4). This mirrors the `reserveReports` precedent (a separate human-owned artifact table beside the machine `runs.recommendations`/`resultSet`), and directly resolves §5.3's deferred "if Epic 6 needs recommendations versioned independently of the Run (draft vs overridden), revisit then."

### §D2 — the first real `requireRole` caller

`requireRole(ctx, workspaceId, "senior_actuary")` exists (`guards.ts:64`) with zero production callers — every reference so far is an anticipatory comment ("override is Epic 6"). 6.3's `overrideRecommendation` is its first use. `requireRole` calls `requireMember` internally (so unauthenticated/cross-workspace callers are rejected at that layer, satisfying the enumeration test) and then throws `FORBIDDEN` when the JWT role ≠ `senior_actuary`. This IS AC-1's "an Analyst's override attempt is rejected server-side" — proven by a convex-test that acts as an analyst identity and asserts `FORBIDDEN` + no row inserted (Task 7.1b). The role lives ONLY in the Clerk JWT (AD-4 — never in a table); the client `canOverride` (D7) is a display convenience, not the authority.

### §D3 — the reason: required, human-owned, audit-carried, not re-gated

FR-10's testable consequence is explicit: "the override and its recorded reason land in the Audit Log." So `recommendation.overridden`'s payload carries the reason verbatim — this is the ONE audit payload in Epic 6 that deliberately includes human prose (the `report.*` audits are lean because the content lives on the row; here the reason's home IS the audit + the override row). The reason is validated for non-emptiness (`REASON_REQUIRED`) — input integrity, NOT the Provenance Gate (AD-5: an override is human-owned, never re-gated; `overrideRecommendation` calls no engine endpoint). AD-1 is not in tension: the reason is the author's professional judgment in words, not an engine-sourced figure, and the mutation does no arithmetic.

### §D4 — the Method picker + re-override semantics

The overriding Method is a `methodValidator` literal OTHER than the recommendation's own method — the picker excludes the recommended method (overriding CL→CL is meaningless) and the server backstops with `OVERRIDE_MATCHES_RECOMMENDATION`. A Senior Actuary may override an already-overridden row (change their mind): each confirm INSERTS a new row (append-only, D1); the side-by-side display shows the LATEST override (max `overriddenAt`) beside the ORIGINAL machine recommendation — never the prior override (the recommendation is the fixed baseline; the current override is the live judgment). Every prior override remains queryable (the query returns all, newest-first) and in the audit log — "history never erased." A full override-history view is a later nicety (deferred-work §6.3).

### §D5 — the per-row status tag is inline, not StatusBadge

`StatusBadge` owns the fixed Run/Report lifecycle vocabulary (UX-DR3, `StatusBadge.tsx:28-36`) — `accepted`/`overridden` are recommendation-ROW states, a different axis, and adding them would overload the component. Render a small inline pill in the Status column using the same token families (`overridden` → amber `bg-caution-subtle text-caution` = "needs your judgment"; `accepted` → neutral `bg-muted text-muted-foreground`), no hard-coded hex (DESIGN.md). Keep `StatusBadge` un-touched.

### §D6/D7 — reuse the audit-dialog primitive; the first current-user-role client signal

6.2 built the `Dialog` primitive + the `DraftBar` audit-dialog contract (Cancel-focus, pending flag, no optimistic UI, inline error) precisely so 6.3/6.4 reuse it — 6.3's override dialog mirrors `DraftBar` exactly (D6). Client role gating is NEW here: today only a member LIST (`seniorActuaries`) is built client-side; no "is the CURRENT user a Senior Actuary" flag exists. 6.3 adds `canOverride = normalizeRole(useAuth().orgRole) === "senior_actuary"` and threads it page → RunDetail → InterpretationTab → RecommendationTable, driving the UX-DR18 visible-but-disabled Analyst control (disabled `<button>` in `<span tabIndex={0}>` + Tooltip "Senior Actuary role required"). This is DISPLAY gating only — the server `requireRole` is the authority (AD-4); the convex-test proves an Analyst call is rejected regardless of what the client renders.

### §D8 — side-by-side, citations intact

AC-2's "recommendation and override display side by side, both attributed, citations intact, history never erased" is a pure display composition over two data sources: the machine `MethodRecommendation` (from `runs.recommendations`, unchanged) and the latest `recommendationOverride` (from the new table). The recommendation's method + reasons + `CitationChip`s render exactly as in 5.5 (the chips are passed the same `dxId`/`diagnosticsBundle` → atomic, navigable, un-re-parsed); the override renders beside it as an attributed card (the `method → method` transition, the quoted reason, `overriddenBy · Senior Actuary · timestamp`) mirroring mockup `report-review.html:64-68` (a teal `border-l` accent — NOT provenance-violet, which stays chip-exclusive, DESIGN.md:89). Nothing is removed on override — the two coexist, which is what "history never erased" looks like in the UI.
