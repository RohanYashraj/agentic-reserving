## Deferred from: Story 4.7 — ResultSet Re-Derivation from Lineage (2026-07-19)

- **Re-derivation compares point estimates only, not the DiagnosticsBundle.** `rederive` re-runs `run_methods` and compares ultimates / IBNR / LDFs / Mack std errs (the AC's "point estimates"); it does not recompute and diff the four Diagnostics. Diagnostics are a deterministic function of the same Triangle + ResultSet, so a reproduced ResultSet implies reproduced Diagnostics — but if a future story wants an explicit Diagnostics re-derivation proof, extend `ReDerivationReport` with a diagnostics section (and add `compute_diagnostics` to the engine `/rederive` path). [engine/reserving_engine/rederivation.py]
- **`rederiveRun` is guarded by `requireMember`, not `requireRole(senior_actuary)` — a deliberate decision.** Re-derivation writes no product state (only an append-only audit entry), so it is a read-only reproducibility proof any Workspace member may run; AD-4 reserves `requireRole` for the approve/publish/**override** *mutation* paths. The story persona is the Senior Actuary (UJ-3/Priya) but the action is not privileged. If the product later wants re-derivation gated to seniors, it is a one-line `requireMember → requireRole(ctx, workspaceId, "senior_actuary")` change plus updating the `authGuard.test.ts` fixture. [convex/runs.ts]
- **The re-derivation verdict lives only in the Audit Log — no `runs` field, no `rederivations` table, no "last verified" badge.** `recordRederivation` appends a `run.rederived` entry and does **not** patch the run row (immutability, AC1); the report is returned to the UI and held in React state (a fresh re-run re-fetches). If a persistent "last re-derived at / verdict" badge is wanted on the Run detail or dashboard, derive it by reading the latest `run.rederived` audit entry for the run (the Epic 7 Audit Log browser already needs to read these) rather than denormalising onto `runs`. [convex/runs.ts]
- **Cross-platform re-derivation uses the documented 1e-8 tolerance tier (`tier: "epsilon"`), exact `==` only on the pinned `linux/x86_64` platform (AD-11).** The tier is reported honestly on the `ReDerivationReport` and surfaced in the UI copy ("Reproduced within 1e-8" vs "Reproduced exactly on the pinned platform"). This is the AD-11 contract, not a silent tolerance widen. The tolerance semantics (`ON_PINNED_PLATFORM`, `_isclose_optional`) are single-sourced in `reserving_engine.rederivation` and imported by `tests/test_rederivation.py`. [engine/reserving_engine/rederivation.py]
- **A down engine surfaces `ENGINE_UNAVAILABLE` to the caller and is NOT recorded as a reproducibility verdict.** The `recordRederivation` audit write is only reached after a successful engine report; a network/5xx failure throws in `callEngine` before the audit mutation, so the Audit Log never carries a false "could not reproduce" entry for an engine outage (distinct from a genuine `reproduced: false`). If operators want engine-outage attempts logged too, add a separate audit event on the catch path. [convex/runs.ts]
